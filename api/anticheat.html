Core — Anticheat server client

The anticheat is native to the core (core/server/anticheat/ + core/client/anticheat/), not a module — it cannot be unloaded from the manifest. It is server-authoritative: the client only reports and polls; every decision (record, kick, ban) is taken server-side, and the client never decides a sanction on its own.

Philosophy​

  • Server-authoritative. Client checks feed the server (lo:anticheat:report, heartbeat, state polls). The server alone records detections and sanctions. Client reports are rate limited (Config.Anticheat.report).
  • Safe by default. Out of the box Config.Anticheat.testMode = true and action = 'log' — the anticheat runs in log-only mode. Detections are persisted to ac_detections and pushed to the Discord webhook, but no player is kicked or banned. This lets you watch real traffic and tune thresholds before enforcing.
  • Countermeasures always run. The "make the exploit not happen" actions — stripping an illegal weapon, deleting a spawned entity, CancelEvent on a forbidden event, dropping a flooded/oversized payload — execute always, even in testMode. Only the player-level sanctions (kick / ban) are suspended while testMode is on. testMode never makes the server more exploitable; it only holds back the punishment.
  • Staff exemption. Players at or above Config.Anticheat.exemptLevel are exempt (Core.Anticheat.IsExempt) — except when forced under watch by the TestKit.

Detections​

High-level list of the vectors covered (server guards consume client reports/game events):

  • Illegal weapon give — a weapon present on the ped but not in the player's inventory is detected on equip/fire; the weapon is stripped and the give is sanctioned.
  • Abnormal weapon damage — per-window damage ceiling and per-hit caps catch inflated damage modifiers / one-shot mods.
  • Aimbot distance — kills landed beyond a per-weapon plausible distance accumulate strikes.
  • AoE / masskill — explosion-spam and clustered deaths in a short window flag area kills.
  • Flood protection — a global per-source event rate guard plus per-event flood guards, with a max-amount guard that catches abusive numeric arguments (e.g. money/item amounts above the rule's ceiling).
  • Forbidden network events — known exploit events (legacy money/item/weapon givers) are blocked and flagged.
  • Wallhack — victim-side raycast confirms line-of-sight before crediting a kill through geometry.
  • Entity-spawn guard — script-spawned peds/objects/vehicles are validated (population type, archetype mismatch, blacklisted/crash models); offenders are removed.
  • Volumetric watchdog — server-side sweep that deletes runaway/orphaned entities above a threshold and isolates the responsible owner.
  • Ped-model swap — a player ped silently becoming a non-player model (animal/prop) is caught.
  • Game events targeting players — weapon/task game events aimed at other players (remote damage, ragdoll, ignite) are flagged.
  • Role-escalation monitoring — unexpected jumps in a player's role/permission state are watched.
  • Heartbeat — the client emits a periodic heartbeat; its absence (AC disabled / unloaded) is detected after a configurable number of missed cycles.
  • Player-state polls — health, invisible, teleport, stamina, freecam, noclip, ped-scale, speed, on-fire/invincible, and cheat-menu global tables are polled and compared against plausible bounds.
  • Stretched resolution — non-16:9 / stretched aspect ratios (a common visibility exploit) are rejected.
  • Resource injection — client resource integrity / injection is monitored, with an optional whitelist and a decoy event that no legitimate client ever triggers.
  • Ban evasion — bans capture multiple identifiers; a reconnecting client matching a banned identifier set is refused.

Configuration​

Everything lives under Config.Anticheat.* in config.lua (and is editable in-game via the framework editor, section Anticheat, which merges over the defaults — see Core.Anticheat.Config). The top-level switches:

KeyDefaultMeaning
enabledtrueMaster switch for the whole anticheat.
action'log''log' (record only), 'kick', or 'ban'.
testModetrueLog-only safety: suspends kick/ban; countermeasures still run.
exemptLevel5Staff level at/above which players are exempt.
checkstablePer-poll toggles (invincible, invisible, speed, pedScale, onfire, freeze, camera, infiniteAmmo, menuGlobals, …).
kickMessage / banMessagestringDrop messages shown to the player.

Each detection family has its own sub-table to tune thresholds — for example weapons, damage, killDistance, explosion, masskill, flood, wallhack, watchdog, entity, state, heartbeat, integrity, banEvasion, extra. Read the inline defaults in config.lua for the full set; they are deliberately conservative.

Going to production​

  1. Stay in testMode first. Run live with testMode = true, watch ac_detections and the webhook, and confirm legitimate play produces no false positives.
  2. Tune the thresholds of any family that flags clean players (raise windows, ceilings, strike counts).
  3. Enforce. Set testMode = false and action = 'kick' (or 'ban'). Countermeasures were already active throughout — this step only turns on player sanctions.

You can flip these at runtime through the editor; Core.Anticheat.Publish() re-broadcasts the effective config (enabled/action/testMode/checks) into GlobalState.Anticheat for the client.

Public API​

Core.Anticheat.Config​

Core.Anticheat.Config()

Returns the effective anticheat config — Config.Anticheat merged with the editor's Anticheat section. server

Returns { enabled, action, testMode, exemptLevel, checks, kickMessage, banMessage }.

Core.Anticheat.Sanction​

Core.Anticheat.Sanction(src, reason, fields, forceBan, checkKey)

Record a detection and apply the effective action. server The call persists the detection to ac_detections, sends the Discord webhook, fires lo:anticheat:detection, then — unless in testMode or action='log' — kicks or bans the player. Exempt staff are ignored. A 30 s anti-spam window per checkKey prevents duplicate sanctions.

NameTypeDescription
srcintegerTarget player.
reasonstringHuman-readable detection reason.
fields?{ name, value, inline? }[]Fields for the webhook embed / persisted detail.
forceBan?booleanBan even if action='kick'.
checkKey?stringAnti-spam key (defaults to reason).
Core.Anticheat.Sanction(src, 'Illegal weapon give', {
    { name = 'Weapon', value = weaponName, inline = true },
}, true, 'weapon_give')

Core.Anticheat.IsExempt​

Core.Anticheat.IsExempt(src) --> boolean

true if the player is exempt (staff level >= exemptLevel), false if they are forced under watch by the TestKit. Always true for an invalid src. server

Core.Anticheat.RateLimit​

Core.Anticheat.RateLimit(name, key, max, window) --> boolean

Sliding-window counter. Returns true once the number of calls for key in bucket name exceeds max within window ms. server

NameTypeDescription
namestringBucket name.
keyanyTracked key (usually src).
maxintegerMax allowed within the window.
windowintegerWindow in milliseconds.
if Core.Anticheat.RateLimit('shop:buy', src, 10, 1000) then return end

Core.SecureEvent​

Core.SecureEvent(name, handler, opts)

Register a hardened server net event. server The wrapper validates the source, applies the abusive-amount guard, enforces an optional minimum staff level, rejects oversized payloads, and rate-limits floods — the handler only runs once everything passes.

OptionTypeDefaultDescription
maxPerSecondinteger20Per-source flood ceiling.
maxBytesintegerConfig.Anticheat.maxEventBytesMax payload size.
minLevelinteger—Minimum staff level (else sanction).
netOnlyboolean—Force RegisterNetOnlyEvent.
Core.SecureEvent('lo:shop:buy', function(itemName, amount)
    -- source is valid, amount checked, payload sane, not flooding
end, { maxPerSecond = 5 })

Audit exports​

Other resources can feed the same pipeline:

exports.lo:ACAudit(src, checkName, details)   -- record + sanction via Core.Anticheat.Audit
exports.lo:Trace(resource, message, data)     -- audit trace into ac_detections

TestKit​

A staff-only in-game tool that triggers every detection vector so you can confirm the anticheat reacts correctly. It is gated and off by default:

  • Enable with Config.Anticheat.testKit = true; access is restricted to Config.Anticheat.testKitLevel (default 50).
  • Open it with the /ac command (or the F10 keybind). It opens an on-screen menu grouped by family: entities/flood, combat/explosions, self (godmode/teleport/noclip…), forbidden events, skin swap, raycast targeting, world chaos, and bypass tests.
  • Arm "Surveillance" to lift your own staff exemption (Core.Anticheat.forceWatched), so the triggers you fire are actually evaluated against you instead of being skipped. /acpanic resets every active toggle.

Use it in testMode to verify that each vector is recorded (and that countermeasures fire), then again after enforcing to confirm kicks/bans land as expected.
