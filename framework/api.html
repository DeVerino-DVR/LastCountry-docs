API du framework ​

Référence des classes et services globaux du framework lo (version 2.0.0, refonte classes/services). Il n'y a plus de table Core : chaque service est un global nommé. Les signatures de cette page reflètent les sources réelles. Les identifiants, events et clés sont en anglais ; les explications sont en français.

Côtés d'exécution

Chaque API précise son côté : (serveur), (client) ou (shared) (disponible des deux côtés). Respecter ces indications évite les erreurs de boot.

Architecture

lo n'a pas de registre de modules, pas d'éditeur in-game, pas de World.Add/Action. Ces systèmes (architecture v1) ont été retirés. Chaque domaine est un dossier plat (server/, client/, shared/) dont les services sont des globaux nommés (classes/services). La config staff passe par la ressource externe lo_jobscreator (voir Config & panels).

Boot & cycle de vie ​

Le framework est initialisé dans init.lua. Les hooks de cycle de vie (shared/lifecycle.lua, shared/event.lua) sont le point d'extension officiel d'un domaine ou d'une ressource tierce.

lua
---@param fn fun()
function Ready(fn) end

Ready (shared, proxy de jo.ready) s'exécute quand le framework est prêt (DB connectée, cache d'items chargé). Différer toute lecture DB ou de Config ici — jamais de MySQL.*.await au top-level d'un fichier (ça avorte le chunk et le service n'est pas exposé).

lua
---@param fn fun(player: LoPlayer)
Lifecycle.OnPlayerLoaded(fn)

---@param fn fun(player: LoPlayer)
Lifecycle.OnPlayerUnloaded(fn)

---@param fn fun(player: LoPlayer)
Lifecycle.OnPlayerSave(fn)

Hooks serveur appelés respectivement après le chargement d'un personnage, à sa déconnexion et à chaque sauvegarde. Chacun reçoit l'instance LoPlayer. Plusieurs hooks peuvent être enregistrés ; ils sont dispatchés dans l'ordre d'enregistrement.

lua
Lifecycle.OnPlayerLoaded(function(player)
    Logger:info('MyDomain', 'Personnage #%s chargé', player.charId)
end)

Machine à états de session ​

CONNECTING → IDENTIFIED → ACCOUNT_LOADED → CHARACTER_SELECT ⟷ CREATOR
   → CHARACTER_LOADING → PRELOADING → SPAWNING → ACTIVE → DROPPING

États définis dans Enums.SessionState (shared/enums.lua). Identité : comptes par fivem: (compte Cfx.re), ciblage stable par userId/charId, jamais par source.

Player serveur ​

Les personnages chargés sont indexés par source dans Players. Chaque entrée est une instance de la classe LoPlayer (player/server/player.lua).

lua

Players = {}

Statebags répliquées ​

Player(src).state reçoit userId, charId, citizenid, job, gang, money, charinfo, onduty (+ vip, adminLevel, IsInSession). Préférer la lecture du statebag (GlobalState + AddStateBagChangeHandler) aux net events pour synchroniser un état.

lua
local player = Players[src]
if player.Functions.HasItem('ticket') and player:RemoveMoney('cash', 50, 'achat-ticket') then
    Inventory.AddItem(src, 'reward', 1)
end

Notifications ​

lua
function Notify(...) end

Notification joueur, signature dépendante du côté.

  • Client : Notify(message, kind, duration?, opts?)
  • Serveur : Notify(src, message, kind?, duration?)

kind vaut 'error', 'success', 'warning', 'info' (défaut 'info'). message peut aussi être une table d'options complète. duration est en millisecondes (défaut 4000 côté client).

lua
-- client
Notify('Action réussie', 'success')

-- serveur
Notify(src, 'Fonds insuffisants', 'error')

Callbacks ​

Callback (shared) est un proxy setmetatable de jo.callback (register / triggerServer / triggerClient / register.latent). C'est le seul aller-retour réseau autorisé — pas d'events manuels.

lua
-- serveur
Callback.register('mydomain:getBalance', function(source, account)
    return Players[source]:GetMoney(account)
end)
lua
-- client (bloquant)
local cash = Callback.triggerServer('mydomain:getBalance', 'cash')

Dans un register serveur, source est le premier argument du handler.

Menu ​

Menu (client) est un proxy setmetatable de jo.menu (create, addItem, show, send, setCurrentMenu, softHide, refresh, isOpen…). L'API jo sous-jacente est identique ; l'indirection par le global nommé est obligatoire.

Commandes & touches ​

Commands:Register ​

lua
---@param name string
---@param fn fun(source:number, args:table, level:number, raw:string)
---@param opts? table
function Commands:Register(name, fn, opts) end

Enregistre une commande (client ou serveur, même signature). Le callback reçoit le niveau staff résolu : serveur Permissions:GetLevel(source), client LocalPlayer.state.adminLevel.

Champ d'optsTypeDescription

Commands:Register('heal', function(source, args, level, raw)
    -- level déjà vérifié si opts.level est défini
end, { level = 'heal', suggestion = 'Soigner un joueur' })

Keys:Register / Keys ​

lua
---@param name string
---@param description string
---@param defaultKey string
---@param category? string
---@param level? number
---@return CoreKeyHandle?
function Keys:Register(name, description, defaultKey, category, level) end

Enregistre une touche remappable pour la ressource courante (jamais RegisterKeyMapping). Le handle expose :onPressed(fn), :onReleased(fn), :onHeld(fn) et :setKey(key). level masque/désactive la touche sous le niveau staff requis. Disponible client. Un export RegisterKey existe également.

lua
Keys:Register('openMenu', 'Ouvrir le menu', 'F5')
    :onPressed(function()
        -- ouverture du menu
    end)

L'API bas niveau Keys (:Register / :On / :Get / :Set / :Reset / :Remove / :GetAll) reste disponible pour les cas multi-ressources, mais Keys:Register est le point d'entrée recommandé.

Events réseau & broadcasts ​

Convention de nommage :

  • Client → Serveur : TriggerServerEvent('lo:Server:<domaine>:<action>', ...), reçu par RegisterNetEvent.
  • Serveur → Client : TriggerClientEvent('lo:Client:<domaine>:<action>', src, ...), reçu par RegisterNetEvent.
  • Tout event traversant le réseau se reçoit avec RegisterNetEvent, jamais AddEventHandler (sinon event X was not safe for net). AddEventHandler uniquement pour un TriggerEvent local (même côté).
  • Aller-retour : Callback.

Net:SafeTriggerClientEvent ​

lua
---@param name string
---@param target integer
---@param ... any
---@return boolean
function Net:SafeTriggerClientEvent(name, target, ...) end

Serveur. Tout broadcast -1 doit passer ici. Au-delà de ~15 Ko de payload, l'envoi bascule en latent (anti-kick 16 Ko qui sinon déconnecte tout le serveur).

lua
Net:SafeTriggerClientEvent('lo:Client:world:refresh', -1, payload)

Net:SecureEvent ​

lua
---@param name string
---@param handler function
---@param opts? table
function Net:SecureEvent(name, handler, opts) end

Serveur. Net event protégé par l'anticheat : opts accepte maxPerSecond, maxBytes, minLevel, netOnly. Rate-limit, taille de payload et niveau requis sont vérifiés avant le handler.

Game events (client) ​

lua
---@param eventName string|number
---@param argCount integer
---@param fn fun(view: DataView)
function GameEvent:On(eventName, argCount, fn) end

Client. Écoute un game-event natif RDR2 (hash ou nom). fn reçoit un DataView des arguments (argCount entiers). Une unique boucle partagée dispatche les handlers — à utiliser plutôt qu'un hook game-event brut.

lua
GameEvent:On('CEventNetworkPlayerEnteredVehicle', 2, function(view)
    local ped = view:GetInt32(0)
end)

Primitives monde (client) ​

Tout spawn/affichage monde passe par ces wrappers de primitives — jamais une native CreatePed/CreateObject/CreateVehicle. Le serveur ne crée jamais d'entité. Streaming par proximité, cleanup et prompts sont gérés.

Primitivedata de :Create (champs clés)
Ped{ id, model, coords, heading?, scenario?, followPlayer?, targetable?, onSpawned? }
Prop{ id, model, coords, heading?, zOffset?, onSpawned? }
Vehicle{ id, model, coords, heading? } (verrouillé, gelé, invincible)
Horse{ id, model, coords, heading?, saddle? } (ped + robe + selle)
Blip{ id, coords, entity?, name?, sprite?, blipHash?, color?, scale?, modifiers?, town?, displayRange? }
Marker{ id, hash, coords, scale?, color?, offset? }
Text3D{ id, text, coords, anchor?, zOffset? }

Chaque primitive expose :Create(data), :Remove(id) et :Get(id). Ped expose aussi Ped:PlayerPedId() et Ped:PlayerId(). Sur un onSpawned(handle), la valeur retournée (ex. un prompt) est détruite au despawn.

Items:Ensure ​

lua
---@param list table[]
function Items:Ensure(list) end

Chaque domaine déclare inline les items dont il a besoin dans son Ready. Ensure prend une liste de définitions (pas (name, def)) et n'insère un item que s'il est absent (jamais d'écrasement). Pas de fichier central.

Champs courants d'une définition : name, label, weight, description, useButton, plus les options avancées (stack, close, consume, degrade, client, server, weapon…). Exports de compat : CreateItem, DeleteItem, GetItem, GetItems.
