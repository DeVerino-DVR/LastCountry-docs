Installation ​

lo est le framework RedM monolithique de Last Country : une ressource unique qui agrège le noyau (sessions, comptes, personnages) et ses modules sous l'objet global Core. Cette page décrit les prérequis serveur, l'ordre de démarrage dans server.cfg, le schéma de base de données auto-géré, et le workflow de développement au quotidien.

Cible RedM uniquement

lo cible RedM / rdr3 exclusivement. Le fxmanifest.lua déclare game 'rdr3' et un rdr3_warning. Ne l'utilisez jamais sur un serveur FiveM.

Prérequis ​

Avant de démarrer lo, votre serveur doit remplir les conditions suivantes, dérivées directement du fxmanifest.lua :

PrérequisDétailSource
FXServerBuild serveur 6231 minimum (/server:6231)dependencies
OneSyncActivé (/onesync)dependencies
oxmysqlRessource oxmysql démarrée avant lodependencies + @oxmysql/lib/MySQL.lua
jo_libsFondation obligatoire (chargée via @jo_libs/init.lua)shared_script
Base MySQL/MariaDBUne base accessible par oxmysql (le schéma est créé automatiquement)jo.database

OneSync est obligatoire

Le framework est serveur-autoritaire et s'appuie sur les routing buckets (sélection de personnage et créateur isolés dans un bucket dédié) ainsi que sur les state bags répliqués (userId, charId, citizenid, job, gang...). OneSync doit être activé pour que ces mécanismes fonctionnent.

jo_libs ​

jo_libs (JollyLib) est la fondation obligatoire de lo : toutes les primitives passent par elle (jo.callback, jo.database, jo.menu, jo.input, jo.player, jo.component, jo.notif, etc.). Le manifeste de lo charge @jo_libs/init.lua en tête des shared_script et active les modules dont il a besoin via les directives jo_lib (player, component, ped-texture, database, menu, input, prompt-nui, dataview, notification, callback, entity, hook, table, utils, blip, game-events, emit, waiter, debugger, nui, animation, raw-keys, prompt).

jo_libs doit être présent et démarré avant lo.

lo fournit (provide) rsg-core et rsg-inventory ​

Le fxmanifest.lua de lo déclare :

lua
provide 'rsg-core'
provide 'rsg-inventory'

Cela signifie que lo se substitue à ces deux ressources : le bridge de lo émet les events miroirs attendus (RSGCore:*) et expose l'objet core via exports['rsg-core']:GetCoreObject(). Les ressources tierces qui dépendent de rsg-core ou rsg-inventory continuent donc de fonctionner sans les ressources d'origine.

Pas de gestion météo dans le core

lo ne fournit pas weathersync : la gestion de la météo et du temps a été retirée du core. Elle est désormais déléguée à une ressource externe dédiée (voir Météo & Temps).

Ne JAMAIS démarrer rsg-core

Comme lo déclare déjà provide 'rsg-core', lancer rsg-core (ou rsg-inventory) en parallèle casse le boot et provoque une double détection du framework. Ces ressources ne doivent jamais figurer dans votre server.cfg. Du code interne à lo, on ne passe d'ailleurs jamais par exports['rsg-core'] ou jo.framework : tout passe par Core.* directement.

Ordre de démarrage dans server.cfg ​

L'ordre est important car lo dépend de oxmysql, de sessionmanager-rdr3 et de jo_libs, charge son NUI depuis lo_uiv2, et parce que toute ressource externe qui inclut un fichier @lo/... (par exemple @lo/bridge/ac_protect.lua) a besoin que lo ait démarré avant elle pour que l'include se résolve.

Ordre recommandé :

  1. oxmysql (la connexion DB doit être prête avant le schéma de lo).
  2. sessionmanager-rdr3 (gestionnaire de sessions RedM).
  3. jo_libs (fondation chargée par lo).
  4. lo_uiv2 (l'interface React, servie par le frame NUI de lo).
  5. lo (le framework, qui fournit rsg-core / rsg-inventory).
  6. Les ressources tierces (chat, anticheat, scripts métier comme lo_jobscreator, ressources externes).

Exemple de bloc server.cfg ​

cfg
# Build serveur et OneSync (prérequis lo)
sv_enforceGameBuild 1491
onesync on

# 1. Base de données
ensure oxmysql

# 2. Gestionnaire de sessions et fondation
ensure sessionmanager-rdr3
ensure jo_libs

# 3. Interface (servie par le frame NUI de lo)
ensure lo_uiv2

# 4. Framework lo
#    fournit rsg-core / rsg-inventory
#    NE JAMAIS ajouter "ensure rsg-core" ici.
ensure lo

# 5. Ressources tierces
ensure chat
ensure lo_ac
ensure lo_jobscreator
# ensure <vos scripts metier>

# Connexion MySQL (oxmysql)
set mysql_connection_string "mysql://user:password@localhost/lastcountry?charset=utf8mb4"

TIP

Si une ressource externe inclut @lo/bridge/ac_protect.lua (protection anticheat de ses entités), elle doit impérativement être démarrée après lo dans le server.cfg pour que l'include @lo/... se résolve.

Schéma de base de données automatique ​

lo ne fournit aucun fichier .sql à importer manuellement. Le schéma est créé et migré automatiquement au démarrage via jo.database.

  • Toute table et toute colonne sont déclarées dans database/server/schema.lua exclusivement, via jo.database.addTable / jo.database.addColumn.
  • Au boot, jo.database crée les tables manquantes et ajoute les colonnes manquantes : un serveur neuf comme une base existante convergent vers le schéma attendu sans intervention.
  • La suppression de personnage est un soft delete (deleted_at), jamais une suppression physique.

Tables principales gérées automatiquement :

sql
users      (id, discord UNIQUE, license UNIQUE, favorite_char_id, banned, created_at)
characters (id, user_id FK, citizenid UNIQUE, slot,
            charinfo JSON, money JSON, job JSON, gang JSON,
            metadata JSON, position JSON,
            skin LONGTEXT, clothes LONGTEXT,
            deleted_at TIMESTAMP NULL,
            created_at, UNIQUE(user_id, slot))

INFO

La seule condition côté DB est qu'oxmysql puisse se connecter à une base existante. Le reste (création des tables, ajout des colonnes) est pris en charge par lo au démarrage.

Workflow de développement ​

lo est du Lua interprété côté serveur : pas de build Lua, pas d'interpréteur local. La boucle de validation est rapide.

Restart à chaud ​

Pour appliquer une modification du code Lua, redémarrez la ressource depuis la console serveur :

bash
restart lo

Le restart en développement est conçu pour être transparent : au onResourceStop, tous les joueurs actifs sont sauvegardés et leur lastCharId conservé dans les state bags (qui survivent au restart de la ressource) ; au onResourceStart, chaque joueur connecté est reconstruit (session relue, re-login automatique sur son dernier personnage) puis respawné en place via Spawner.spawn{ reason = 'restart' }. Résultat : tout le monde est rechargé sans écran de sélection ni téléportation, skin réappliqué.

Avant de livrer

Aucun interpréteur local n'est disponible : validez toujours par restart lo + lecture de la console serveur + test côté client (F8). Vérifiez l'équilibre syntaxique (parenthèses, end, accolades) avant de redémarrer.

Console client (F8) et logs ​

  • Ouvrez la console client avec F8 en jeu pour observer les erreurs et messages côté client.
  • Côté serveur, surveillez la console qui trace tout le cycle de vie de la session (session:transition) et les logs des modules.

Le logger est piloté par catégorie via Core.Logger :

lua
Core.Logger.info('Catégorie', 'message...')

Les catégories sont activables/désactivables par la table Config.Debug et, à chaud, par la commande :

bash
/debug <Catégorie> <true|false>

Front (lo_uiv2) ​

Le front React vit dans une ressource séparée, lo_uiv2. Après une modification de l'interface :

bash
cd resources/[ui]/lo_uiv2/web
npm run build

Puis, depuis la console serveur, redémarrez uniquement l'interface (inutile de redémarrer lo) :

bash
restart lo_uiv2

Autocomplétion VSCode ​

Pour développer contre l'API du framework, installez l'extension VSCode communityox.cfxlua-vscode-cox (natives RedM). Les définitions de l'API publique de lo (Core.*, Skin) sont fournies par types/core.lua (annotations LuaLS ---@meta), à tenir à jour à chaque évolution d'API.
