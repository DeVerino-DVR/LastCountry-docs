Bridge RSG ​

Le bridge est la couche de compatibilité qui permet à des ressources RSG externes de tourner sur lo sans modification. lo n'embarque pas de cœur RSG : il l'implémente. Le framework déclare provide 'rsg-core' et provide 'rsg-inventory' dans son fxmanifest.lua, expose l'export GetCoreObject() attendu par l'écosystème RSG, et miroite les events lo:* vers RSGCore:* (et inversement) aux bons moments du cycle de vie. Une ressource écrite pour RSG appelle ses APIs habituelles ; le bridge les redirige vers Core.

Garde-fou absolu

rsg-core ne doit jamais être démarré en parallèle de lo. Le fxmanifest.lua de lo contient déjà provide 'rsg-core' et provide 'rsg-inventory' : c'est lo qui fournit ces ressources. Démarrer un vrai rsg-core à côté provoque un conflit d'exports et casse le boot.

Pourquoi un bridge ​

Les ressources RSG consomment trois choses de leur cœur :

  • l'objet core (exports['rsg-core']:GetCoreObject()) et ses sous-tables Functions, Shared, Player, PlayerData ;
  • les events de cycle de vie (RSGCore:Client:OnPlayerLoaded, RSGCore:Server:OnPlayerUnload, mises à jour de job/argent…) ;
  • l'API d'inventaire (exports['rsg-inventory']) pour les stashs, l'ajout/retrait d'items, etc.

Le bridge fournit ces trois surfaces en s'adossant à l'API native de lo (PlayerService:*, Inventory.*, Callback.*, Shared.*). Aucune logique métier n'est dupliquée : le bridge n'est qu'un adaptateur de signatures.

Usage interne

Le bridge ne sert que les ressources externes. Le code de lo n'appelle jamais GetCoreObject() ni jo.framework/jo.fw : en interne tout passe par Players[src].Functions.*, Inventory.*, Sessions et les statebags. Le provide rsg-core de lo étant détecté par le framework-bridge de jo_libs, l'utiliser depuis lo serait un aller-retour qui casse le boot.

exports['rsg-core']:GetCoreObject() renvoie une table (l'objet RSGCore construit par le bridge) avec les sous-tables Functions, Player, Shared, Config.

CreateCallback adapte la convention RSG (callback à deux temps cb(source, done, ...)) vers le modèle de Callback qui retourne directement des valeurs :

lua
local Core = exports['rsg-core']:GetCoreObject()

Core.Functions.CreateCallback('myresource:getBalance', function(source, cb)
    local player = Core.Functions.GetPlayer(source)
    cb(player.PlayerData.money.cash)
end)

Côté appelant client, TriggerCallback reste classique :

lua
local Core = exports['rsg-core']:GetCoreObject()

Core.Functions.TriggerCallback('myresource:getBalance', function(cash)
    print('cash =', cash)
end)

GetCoreObject() — surface client ​

Côté client, exports['rsg-core']:GetCoreObject() renvoie une table plus légère :

MembreComportement
Functions.GetPlayerData()retourne PlayerData.current (les données du joueur local)
Functions.TriggerCallback(name, cb, ...)déclenche un callback serveur via Callback.triggerServer

La liste d'items client est récupérée au boot via un callback, puis figée :

lua
local items = Callback.triggerServer('lo:inventory:getItemList')

Deux exports de version sont fournis pour compat : GetCoreVersion() retourne 'lo' (client et serveur).

Format PlayerData (aligné RSG) ​

L'objet Player de lo est volontairement structuré pour correspondre aux attentes RSG : la clé d'interop est citizenid, sur laquelle toutes les ressources RSG indexent. La structure répliquée au client est la suivante.

lua
PlayerData = {
    citizenid = string,
    license = string,
    discord = string,
    charinfo = { firstname, lastname, birthdate, gender, nationality },
    money = { cash = 0.0, gold = 0.0 },
    job = { name = 'unemployed', label = 'Sans emploi', grade = { level = 0, name = 'none' }, onduty = false, payment = 0 },
    gang = { name = 'none', label = 'Aucun', grade = { level = 0, name = 'none' } },
    metadata = { hunger = 100, thirst = 100, stress = 0, isdead = false },
    position = vector4,
}

PlayerData est répliqué au client à chaque mise à jour via lo:Player:SetPlayerData, que le bridge relaie en RSGCore:Player:SetPlayerData. Une ressource RSG lit donc Core.Functions.GetPlayer(src).PlayerData.citizenid exactement comme elle le ferait sur un vrai cœur RSG.

Mirroring des events lo:* ↔ RSGCore:* ​

Le bridge installe des relais bidirectionnels : tout event lo:<name> est répercuté en RSGCore:<name> et inversement. Un verrou (active) empêche les boucles infinies de réémission.

lua
local function link(from, to)
    AddEventHandler(from, function(...)
        if active[from] then return end
        active[to] = true
        TriggerEvent(to, ...)
        active[to] = false
    end)
end

Le point clé : ce sont les events émis par le pipeline de lo aux bons moments du cycle de vie qui sont miroités. Le bridge ne fabrique pas d'event artificiel — il relaie l'event que la session a déjà déclenché à l'étape exacte (par exemple activate pour OnPlayerLoaded, le logout pour OnPlayerUnload).

Cas particulier : OnPlayerLoaded / OnPlayerUnload serveur ​

Server:OnPlayerLoaded et Server:OnPlayerUnload sont des events qui doivent partir avec la source comme argument et atteindre le bon contexte. Le bridge gère le routage : si le joueur est toujours connecté, le serveur demande au client de réémettre l'event via un écho contrôlé (lo:bridge:echo) ; sinon il l'émet localement.

lua
AddEventHandler('lo:Server:OnPlayerLoaded', function(src)
    src = tonumber(src)
    if not src or src <= 0 then return end
    if GetPlayerName(src) then
        TriggerClientEvent('lo:bridge:echo', src, 'RSGCore:Server:OnPlayerLoaded')
    else
        TriggerEvent('RSGCore:Server:OnPlayerLoaded', src)
    end
end)

Côté client, seuls deux noms d'event sont autorisés à être réémis vers le serveur par l'écho, pour éviter qu'un client n'injecte n'importe quel event serveur :

lua
local ECHO_ALLOWED = {
    ['RSGCore:Server:OnPlayerLoaded'] = true,
    ['RSGCore:Server:OnPlayerUnload'] = true,
}

Le format d'un slot retourné est aligné RSG (les deux clés coexistent pour la compatibilité) :

lua
item.amount = slotItem.count
item.info = slotItem.metadata or {}

Le bridge ponte aussi les events de fermeture d'inventaire dans les deux sens, en couvrant les variantes de casse RSG :

lua
RegisterNetEvent('rsg-inventory:client:closeInv', function() Inventory.Close(true) end)
RegisterNetEvent('rsg-inventory:client:closeinv', function() Inventory.Close(true) end)

AddEventHandler('lo:inventory:closed', function()
    TriggerEvent('rsg-inventory:client:closeinv')
    TriggerEvent('rsg-inventory:client:closeInv')
end)

Proxys d'exports ​

Au-delà des exports déclarés directement, le bridge intercepte les demandes d'export de rsg-core et rsg-inventory via le mécanisme __cfx_export_*. Cela couvre les ressources qui résolvent un export au démarrage.

Côté serveur, les exports rsg-core proxifiés incluent :

lua
'GetCoreObject', 'SetMethod', 'SetField', 'AddJob', 'AddJobs', 'RemoveJob', 'UpdateJob',
'AddGang', 'AddGangs', 'RemoveGang', 'UpdateGang', 'ExploitBan'

Côté client, les exports rsg-core proxifiés incluent DrawText, ChangeText, HideText, KeyPressed, plus GetCoreObject et GetCoreVersion. Chaque proxy redirige vers l'export lo correspondant :

lua
local function makeProxy(name)
    return function(...)
        return exports.lo[name](nil, ...)
    end
end

Un export serveur ExploitBan(src, reason) est également fourni : il DropPlayer la source avec la raison donnée.

Audit depuis une ressource externe ​

bridge/audit.lua définit deux helpers serveur sur la table globale Lo, à utiliser depuis une ressource externe pour remonter une détection ou une trace vers lo. Les deux sont gardés par l'état de la ressource lo et un pcall.

HelperEffet
Lo.Audit(src, checkName, details)appelle exports.lo:ACAudit(src, checkName, details)
Lo.Trace(message, data)appelle exports.lo:Trace(GetCurrentResourceName(), message, data)

Protéger les entités d'une ressource externe de l'anticheat ​

L'anticheat de lo surveille les entités créées côté client (flood de création, comptage par propriétaire avec suppression). Une ressource légitime qui spawn beaucoup d'entités (chevaux, props, véhicules) doit signaler ses spawns pour ne pas se faire supprimer ses entités. Le mécanisme @lo/bridge/ac_protect.lua permet cela sans toucher au code de la ressource externe.

Ajouter le script en tête des client_scripts du fxmanifest.lua de la ressource externe :

lua
client_scripts {
    '@lo/bridge/ac_protect.lua',
    -- ... le reste des scripts de la ressource
}

ac_protect.lua wrappe les natives de spawn dans le contexte Lua de la ressource et tague chaque entité créée d'un state bag répliqué lo_protected = true (avec un décorateur DecorSetBool en doublon). Les natives wrappées :

lua
CreatePed   CreateObject   CreateObjectNoOffset   CreateVehicle
CreateMissionTrain   CreatePedInsideVehicle

Le tag s'applique uniquement si l'entité existe, et peut être désactivé temporairement par le compteur global LO_AC_NO_TAG.

lua
local function tag(entity)
    if (_G.LO_AC_NO_TAG or 0) > 0 then return entity end
    if entity and entity ~= 0 and DoesEntityExist(entity) then
        pcall(function() Entity(entity).state:set('lo_protected', true, true) end)
        pcall(function() DecorSetBool(entity, 'lo_protected', true) end)
    end
    return entity
end

Côté serveur, le watchdog ignore toute entité où Anticheat.IsEntityProtected(ent) est vrai : elle n'est ni comptée, ni supprimée (le nettoyage de flood réutilise ce même scan).

Ordre de chargement

ac_protect.lua n'a aucune dépendance au runtime de lo, donc aucun souci d'ordre d'exécution interne. En revanche lo doit démarrer avant la ressource externe dans server.cfg pour que l'include @lo/... se résolve. Ce fichier est déjà listé dans le files {} du fxmanifest.lua de lo ; sans cela l'include échouerait (Failed to load script @lo/bridge/ac_protect.lua).

Limite assumée

lo_protected est un state bag posé côté client, donc spoofable par un cheateur sur ses propres entités. Les modèles de crash et blacklistés restent supprimés quoi qu'il arrive : le flag ne relâche que l'heuristique de comptage. Pour durcir avant un passage en mode actif, valider les modèles autorisés côté serveur.
