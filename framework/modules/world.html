Primitives monde (world/) ​

Le domaine world/ ne fournit que des primitives client de matérialisation : des wrappers qui créent et streament des entités par proximité, et qui attachent des prompts d'interaction. Il n'y a pas de registre serveur d'entités, pas de table world_entities, pas de synchronisation d'état monde, et pas de panel /worldedit : ces systèmes (architecture v1) ont été retirés.

Les interactions de jeu (boutiques, coffres, métiers, ateliers, livraisons, etc.) ne sont pas gérées par le core. Elles sont définies et placées via la ressource externe lo_jobscreator (types d'interaction custom), qui appelle les primitives ci-dessous pour s'afficher et relaie les actions au serveur. Voir Éditeur de configuration.

Le serveur ne crée jamais d'entités

Toute entité de jeu (ped, prop, véhicule, blip, marqueur…) est créée uniquement côté client par les wrappers Core.*. Le serveur ne matérialise rien. N'appelez jamais CreatePed, CreateObject ou CreateVehicle directement : passez toujours par Core.Ped, Core.Prop, Core.Vehicle, etc. Ces wrappers gèrent le streaming, le nettoyage et la cohérence.

Vue d'ensemble ​

Chaque primitive vit dans world/client/ et expose Create(data), Remove(id) et Get(id). Toutes prennent un champ id (clé de gestion interne) et un champ de position coords (table x/y/z ou vector3).

Groupage par ville

Si Config.BlipGrouping.enabled est vrai, un blip sans town est rattaché à la ville la plus proche dans le rayon Config.BlipGrouping.autoTownRadius.

Core.Ped ​

Core.Prop et Core.Vehicle ​

Mêmes principes de streaming par proximité que Core.Ped.

lua
Core.Prop.Create({
    id = 'stall_1',
    model = 'p_market01x',
    coords = { x = -278.0, y = 805.0, z = 119.0 },
    heading = 180.0,
    zOffset = 0.0,
    prompt = { label = 'Ouvrir', key = 'G', distance = 2.0, onUse = function(obj, id) end },
})

Core.Vehicle.Create({
    id = 'wagon_1',
    model = 'wagon02x',
    coords = { x = -300.0, y = 790.0, z = 118.0 },
    heading = 0.0,
})

Le prop accepte onSpawned et zOffset. Le véhicule est figé, invincible et verrouillé.

Prompt clic-droit sur un prop : focus targetable

Les objets ne répondent pas au weapon-target. Pour un prompt de ciblage sur un prop, la primitive active le focus targetable (SetObjectTargettable + le natif _SET_OBJECT_TARGETTABLE_FOCUS, hash 0xA22712E8471AA08E) au spawn. Le ciblage se résout ensuite via GetPlayerInteractionTargetEntity. Il suffit de passer un champ prompt (ou targetable = true) à Core.Prop.Create — la primitive s'en charge.

Core.Marker et Core.Text3D ​

lua
Core.Marker.Create({
    id = 'shop_marker',
    coords = { x = -276.0, y = 804.0, z = 119.0 },
    hash = 0x6903B113,
    scale = 1.0,
    color = { r = 255, g = 200, b = 0, a = 150 },
    offset = 0.0,
})

Core.Text3D.Create({
    id = 'shop_label',
    coords = { x = -276.0, y = 804.0, z = 119.0 },
    text = 'Magasin',
    anchor = 'point',
    zOffset = 1.0,
})

Le texte 3D peut s'ancrer sur l'entité gérée du même id : anchor = 'ped' (suit l'os de tête) ou anchor = 'prop' (au sommet du prop). Les deux primitives rendent uniquement dans Config.Performance.markerRenderDistance.

Core.Prompt ​

Core.Prompt expose les constructeurs de prompts (les touches passent par Core.PromptKeys) :

FonctionUsage
Core.Prompt.createNative(coords, label, distance, keyName, callback)Prompt fixe à des coordonnées
Core.Prompt.createTarget(entity, label, name, distance, callback, keyName)Prompt de ciblage sur une entité
Core.Prompt.createFollow(getEntity, label, distance, keyName, callback)Prompt suivant une entité dynamique

Chaque constructeur renvoie une instance avec :Destroy() (et :SetLabel(text) pour le prompt « follow »).

Core.Interact (composition) ​

Core.Interact assemble plusieurs primitives sous un seul id, et nettoie tout d'un coup au Remove. C'est la brique pratique pour matérialiser un point d'interaction complet.

lua
Core.Interact.Create({
    id = 'shop_point_1',
    coords = { x = -276.0, y = 804.0, z = 119.0 },
    blip = { sprite = 'blip_ambient_new', name = 'Magasin' },
    marker = { color = { r = 255, g = 200, b = 0, a = 150 } },
    ped = { model = 'cs_marshall', heading = 90.0 },
    text = 'Magasin',
    prompt = { label = 'Ouvrir', key = 'G', distance = 2.0 },
    onSelect = function(id) end,
})

Règles de composition :

  • blip et marker héritent des coords de l'interaction s'ils n'ont pas les leurs.
  • ped ou prop (exclusifs) ; si un prompt est fourni, il est attaché en prompt de ciblage sur l'entité au spawn.
  • text crée un Core.Text3D ancré automatiquement (ped/prop/point selon l'entité présente).
  • Sans ped ni prop, le prompt devient un prompt natif aux coords.

Interactions de jeu — lo_jobscreator ​

Les boutiques, coffres, métiers et autres points d'interaction du serveur sont créés par la ressource externe lo_jobscreator, qui consomme ces primitives côté client. Le bridge bridge/server/jobscreator.lua enregistre les filtres reliant lo_jobscreator au cœur de lo :

  • catalogue d'items (items:list, items:create, items:update, items:delete) depuis Core.Items ;
  • ouverture de coffre (inventory:open), de boutique (shop:open) et d'établi (craft:open) via Core.Inventory ;
  • chemin des images items (items:imageUrl → nui://lo_uiv2/web/images/%s.png).

lo_jobscreator expose aussi des types d'interaction custom : exports.lo_jobscreator:RegisterInteractionType{...} / RegisterInteractionFields{...}. Toute logique d'interaction monde se définit là, pas dans le core.
