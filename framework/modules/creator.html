Module Créateur de personnage ​

Le domaine creator est le créateur et l'éditeur d'apparence du framework lo. Il enchaîne un workflow par étapes (identité, apparence, visage, cheveux, tenue, revue, validation), applique chaque changement en direct sur le ped via la couche Skin.*, et expose des outils staff en jeu pour extraire et gérer les catalogues de composants. Le créateur fait partie du domaine character : toute la logique vit dans character/client/creator/, et les systèmes réutilisables (caméra, données d'apparence) restent dans le même domaine.

Cette page documente ce qui existe réellement dans le code : les wrappers CreatorNatives.*, l'orchestration des menus, les scènes caméra, les catalogues de données, les commandes staff et la validation serveur.

Globaux nommés

L'API publique passe par des globaux nommés (Skin, Menu, Notify, CharacterData…). Les fichiers du créateur exposent en plus quelques globales locales au resource (CreatorNatives, CreatorCamera, CreatorScene) consommées entre eux.

Arborescence ​

Le créateur est entièrement côté client, à l'exception de la logique serveur (dans le domaine character) pour les buckets, la blacklist et la sauvegarde des extractions.

character/client/creator/
├── api.lua          CreatorNatives.* (wrappers natives metaped)
├── camera.lua       CreatorCamera.* (scène studio, rotation, zoom, prompts)
├── scene.lua        CreatorScene.* (bucket, imap, intérieur, lumières)
├── extractor.lua    /extract, /dedupe (outils staff)
├── workflow.lua     orchestration des menus + flux create/edit
├── identity.lua     saisie d'identité (NUI)
├── spawn.lua        sélection de ville + kit de départ
└── data/            catalogues statiques (labels, hairs, exclusive, extracted_*, ...)

Le workflow par étapes ​

Le flux complet va de la saisie d'identité jusqu'à l'insertion serveur du personnage.

1. Identité      input NUI : prénom, nom, naissance, nationalité, genre
2. Apparence     corps : visage, teint, carrure, taille
3. Visage        yeux, sourcils, nez, bouche, structure
4. Cheveux       cheveux, barbe / moustache / favoris / bouc (selon le genre)
5. Tenue         maquillage & défauts, éléments exclusifs par catégorie
6. Revue         rotation libre du ped, application live, retour à toute étape
7. Validation    envoi serveur → validation des données → INSERT + citizenid
8. Login + Spawn rejoint le pipeline standard (sélection de ville, kit de départ)

Deux modes partagent le même code de menus (mode = 'create' ou 'edit') :

EntréeModeDéclencheur
lo:creator:start (net event)createnouveau personnage, identité demandée puis ville et kit de départ
lo:creator:edit (net event) / commande /skineditédition de l'apparence du personnage chargé, sauvegarde directe

Le bouton final du menu racine appelle validate() :

lua
local function validate()
    if mode == 'create' then
        Menu.show(false)
        Wait(250)
        identity = askIdentity()
        if not identity then
            show('lo_creator_main')
            return
        end
        openSpawnSelect()
    else
        TriggerServerEvent('lo:server:saveCharacter', editing, buildCleanClothes())
        Notify('Apparence sauvegardée', 'success')
        finishCreator(true)
    end
end

En mode create, l'identité est saisie en dernier (NUI), puis le joueur choisit sa ville (openSpawnSelect) et son kit de départ (openStarterPackSelect) avant l'envoi de lo:server:createCharacterWithSkin. En mode edit, la sauvegarde est immédiate.

État d'édition ​

L'état courant vit dans deux tables locales :

  • editing — toutes les valeurs d'apparence (indices, teintes, expressions, overlays, échelle…). defaultEditing() fournit les valeurs initiales.
  • clothes — les vêtements portés par catégorie.

buildCleanClothes() sérialise clothes en une table propre (drawable + albedo/normal/material/palette/teintes, ou simplement hash + palette/teintes) avant l'envoi serveur.

Étape identité (NUI) ​

L'identité passe par une fenêtre NUI dédiée, pas par un menu jo.menu. askIdentity() boucle tant que les champs sont invalides :

RegisterNUICallback('identity:submit', function(data, cb) cb('ok') ... end)
RegisterNUICallback('identity:cancel', function(_, cb) cb('ok') ... end)
RegisterNUICallback('identity:sound', function(data, cb) ... end)

CreatorNatives.* — wrappers metaped ​

client/api.lua regroupe les natives metaped brutes derrière des wrappers nommés. Elles sont chargées tôt et consommées par l'extracteur.

WrapperNativeRetour
CreatorNatives.GetNumComponentsInPed(p)0x90403E8107B60E81nombre de composants du ped (ou 0)
CreatorNatives.GetCategoryOfComponentAtIndex(p, componentIndex)0x9B90842304C938A7hash de catégorie du composant
CreatorNatives.GetMetaPedAssetGuids(p, index)0xA9C28516A6DC9D56drawable, albedo, normal, material
CreatorNatives.GetMetaPedAssetTint(p, index)0xE7998FEC53A33BBEpalette, tint0, tint1, tint2
CreatorNatives.GetShopItemBaseLayers(hash, metapedType)0x63342C50EC115CE8drawable de base d'un shop item

Piège vérifié — GetMetaPedAssetGuids retourne 4 valeurs directes

GetMetaPedAssetGuids et GetMetaPedAssetTint renvoient directement 4 valeurs (les pointeurs passés via Citizen.PointerValueInt()), sans retval booléen en tête. C'est pourquoi readPart lit drawable, albedo, normal, material dès la première valeur de retour.

lua
local drawable, albedo, normal, material = GetMetaPedAssetGuids(p, index)
if not drawable or drawable == 0 or not albedo or albedo == 0 then return nil end
local palette, tint0, tint1, tint2 = GetMetaPedAssetTint(p, index)

GetShopItemBaseLayers ne renvoie rien d'exploitable pour les catégories bodies_* : pour ces composants, on lit les couches sur le ped porté plutôt qu'à partir d'un shop item.

La scène caméra (CreatorScene) ​

client/scene.lua met en place le studio isolé où l'édition a lieu.

CreatorScene.Enter(keepOrigin) :

  1. Fond noir (DoScreenFadeOut), puis TriggerServerEvent('lo:creator:enterBucket') pour s'isoler.
  2. Téléporte le ped sur Config.Game.creatorManager (vector4 position + heading) et le fige.
  3. Charge les imaps Config.Game.creatorImaps et active les entity sets de Config.Game.creatorInterior.
  4. Force le streaming autour de la scène (SetFocusPosAndVel, RequestCollisionAtCoord, LoadSceneStart) avec un délai borné par Config.Creator.SceneLoadTimeout.
  5. Masque la minimap et l'UICore (exports.lo:setVisible(false)), puis monte l'éclairage studio.

CreatorScene.GroundPed() repose le ped au sol après un changement de taille ou de carrure, puis le refige. CreatorScene.Exit(restoreOrigin) défait tout (désactive les entity sets, retire les imaps, restaure radar et UI, replace le ped si keepOrigin avait été demandé) et sort du bucket.

Référence des patterns UI

Le créateur est la référence d'UI riche du framework. Pour toute nouvelle UI : préview au onActive, application par prompt ENTER maintenu, coche iconRight='tick' sur l'élément porté, item « Aucun » avec préview du retrait, restauration de l'état réel au retour sans validation, et pour les grosses listes : construction différée au clic + pagination 80 + regroupement par origine.

Corps et silhouette ​

openBody() règle le visage (modèle 1-20), le visage exclusif (staff/VIP), le teint de peau (sprites 1-6), la carrure (1-5) et la taille (bodyScale 0.60-1.20, via Skin.SetScale puis CreatorScene.GroundPed). openSilhouette() expose des morphs continus de carrure (bras, épaules, omoplates, poitrine, taille/hanches, jambes, cou) appliqués par Skin.ApplyExpression.

Visage ​

openFace() regroupe les sous-écrans yeux, sourcils, nez, bouche et structure. Chaque morph 1D ou 2D est un curseur en grille mappé sur [-1.0, 1.0] qui écrit dans editing.expressions[key] et appelle Skin.ApplyExpression(ped(), key, value) en direct.

Piège vérifié — animation faciale de la bouche

L'écran « Bouche » ouvre la mâchoire du ped pour montrer les dents avec l'override facial Face_Dentistry_Loop (dictionnaire FACE_HUMAN@GEN_MALE@BASE). Il faut impérativement refermer la bouche (Face_Dentistry_Out) en quittant l'écran. Un thread de garde (startMouthGuard) surveille currentMenuId et referme la bouche dès qu'on n'est plus sur lo_creator_mouth, pour éviter qu'un ped reste figé bouche ouverte.

lua
local function playFaceAnim(anim)
    if anim == 'Face_Dentistry_Loop' then
        mouthOpen = true
        ensureFaceDict()
        SetFacialIdleAnimOverride(ped(), 'Face_Dentistry_Loop', FACE_DICT)
    else
        mouthOpen = false
        setNeutralFace()
    end
end

Cheveux et barbe ​

openHair() propose les cheveux et (pour mp_male) la barbe à partir de CharacterData.Hairs[sexKey()], avec palette de couleur dédiée. Il branche ensuite les catalogues exclusifs par catégorie via addExclusiveItem :

GenreCatégories exclusives proposées
mp_malehair, beards_complete, beards_mustache, beards_chops, beards_chin
mp_femalehair, hair_bonnet

Maquillage et overlays ​

openMakeup() ajoute des overlays via addOverlay (vieillissement, cicatrices, taches de rousseur, grains de beauté, acné, teint, saleté, et le maquillage teintable : ombre à paupières, blush, rouge à lèvres, eyeliner, fond de teint). Chaque overlay a un type, une opacité et, pour le maquillage, une palette de teintes. Les valeurs vivent dans editing.overlays[category] et sont appliquées par Skin.ApplyOverlay / Skin.RemoveOverlay.

Palettes de couleur ​

openColorPalette(opts) est l'écran de teintes partagé. Il liste les palettes disponibles (libellées par CharacterData.Labels.PaletteNames), borne chaque curseur par CharacterData.Labels.PaletteSizes, et applique les 3 teintes via le callback set(tints, palette) du contexte. Le préfixe moteur metaped_ est ajouté automatiquement (paletteEngine).

Mode debug, blacklist, recatégorisation ​

En mode debug (F5), chaque élément exclusif peut être :

  • caché aux joueurs (X) : lo:creator:toggleBlacklist côté serveur écrit data/creator_blacklist.json puis rebroadcast lo:creator:blacklistUpdated. Les menus ouverts se rafraîchissent en place (activeCatalogRefresh).
  • déplacé de catégorie (C) : lo:creator:setCategory écrit data/creator_overrides.json puis rebroadcast lo:creator:overridesUpdated. La clé visée a la forme exclusive:<sexe>:<cat>:<drawable>.

La commande console/staff /creatorblacklist liste les éléments cachés ; creatorblacklist clear réinitialise la blacklist.

/extract — extraction des composants ​

/extract <catégorie|all> [préfixe] (staff niveau 40) instancie les peds de CharacterData.ExtractPeds, parcourt leurs tenues et collecte les composants par catégorie à l'aide des CreatorNatives.*.

/extract all            extrait toutes les catégories
/extract heads          extrait uniquement les visages
/extract coats cs_      limite aux peds dont le nom commence par cs_

Détails du fonctionnement :

  • Extraction parallèle sur WORKERS = 8 threads.
  • Pour les catégories listées dans WITH_BODY (visages, manteaux, hauts, robes, pantalons…), le corps porté (bodies_upper / bodies_lower) est lu et attaché à l'élément.
  • Chaque catégorie est renvoyée au serveur via TriggerLatentServerEvent('lo:creator:extractSave', 200000, catName, content).
  • Côté serveur, lo:creator:extractSave revalide le niveau et le nom de catégorie, puis écrit character/client/creator/data/extracted_<cat>.lua.

/dedupe — retrait des doublons multijoueur ​

/dedupe (staff niveau 40) scanne tous les shop items multijoueur via GetShopItemBaseLayers, repère leurs drawables, puis retire des catalogues exclusifs (CharacterData.Exclusive) toute entrée dont le drawable correspond à un vêtement MP déjà disponible. Les catégories modifiées sont réécrites via lo:creator:extractSave.

Les commandes sont suggérées dans le chat au démarrage du resource (chat:addSuggestion).

Régénération des catalogues

Les catalogues se régénèrent en jeu : /extract all (staff 40), puis post-traitement des noms et couleurs (scripts externes), puis /dedupe.

Isolation par bucket (serveur) ​

character/server/creator.lua gère l'isolation du créateur. À l'entrée dans le créateur, le joueur est déplacé dans un routing bucket aléatoire avec population désactivée ; à la sortie, il revient au bucket 0.

lua
Net:SecureEvent('lo:creator:enterBucket', function()
    local src = source
    if not Sessions[src] then return end
    local bucket = math.random(100000, 999999)
    SetPlayerRoutingBucket(tostring(src), bucket)
    SetRoutingBucketPopulationEnabled(bucket, false)
end, { maxPerSecond = 3 })

Broadcasts protégés

Les rebroadcasts lo:creator:blacklistUpdated et lo:creator:overridesUpdated passent par Net:SafeTriggerClientEvent, qui bascule en latent au-delà de ~15 Ko pour livrer un gros payload sans kick.

Validation serveur de la création ​

Le serveur ne fait jamais confiance au payload client. À la création (lo:server:createCharacterWithSkin), chaque champ est revalidé avant l'INSERT.

DonnéeValidation serveur
firstname / lastnamesanitizeName (rejet si vide/invalide)
birthdatesanitizeBirthdate
nationalitysanitizeName
cityKeyresolveCitySpawn (ville inconnue → refus)
skin / clothesdoivent être des tables
starterPackKeyresolveStarterPack

Le genre est dérivé du skin (skin.model == 'mp_female' → female), le personnage est créé via Character:Create, défini comme favori, puis le joueur est rebasculé dans le pipeline de login/spawn. La sauvegarde d'apparence en mode édition (lo:server:saveCharacter) vérifie l'existence du Player et le type des tables avant Character:SaveAppearance.
