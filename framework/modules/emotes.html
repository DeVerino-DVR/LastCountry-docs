Emotes ​

Le module emotes gère l'ensemble des animations jouables par le joueur : émotes classiques, émotes avec objets (props), danses, danses synchronisées, émotes KIT natives, scénarios, animations de cheval, styles de marche, vitesse de marche, styles de combat à mains nues, expressions faciales et décals corporels. Toutes les données sont définies côté client dans client/data/, indexées au démarrage, et exposées via l'objet global Emote. Le menu intégré (EmoteMenu) et un jeu de commandes chat permettent de déclencher chaque action.

L'API n'est pas exposée via exports : elle s'appuie sur des globaux nommés. Emote (jouer/annuler) et EmoteMenu (interface) sont des classes globales directes, et EmoteData / EmoteLookup sont des tables globales (données et index).

Configuration ​

La configuration vit dans config/config.lua, sous Config.Emotes.

ChampTypeDescription
AdminLevelnumberNiveau de permission requis pour les émotes marquées isAdmin et pour bypasser les requêtes de synchronisation.
VipMinRanknumberRang VIP minimum pour les émotes marquées isVip.
openKeystringTouche par défaut d'ouverture du menu (F9).
cancelKeystringTouche par défaut d'arrêt de l'animation (X).
favoriteKeystringTouche pour mettre en favori l'émote survolée dans le menu (E).
menuRowsnumberNombre de lignes affichées par page de menu.
syncEmoteDistancenumberDistance maximale (côté client) pour trouver un partenaire d'émote synchronisée.
syncRequestTimeoutnumberDélai d'acceptation d'une requête de synchronisation, en millisecondes.
nearbyScenarioRadiusnumberRayon de détection utilisé par les scénarios à proximité.
defaultWalkRationumberRatio de marche par défaut.
WalkSpeedstableListe de presets { label, ratio } pour la vitesse de marche.
PreviewtableRéglages caméra de prévisualisation (distance, side, height).

INFO

Les contrôles d'accès s'appuient sur d'autres domaines : Permissions:HasLevel côté serveur, Vip:HasTier / Config.Vip.Tiers pour les rangs VIP.

Catégories de données ​

Chaque fichier de client/data/ enrichit la table EmoteData. Les catégories d'émotes (General, Props, Dances, Kit, Synchronized, Horse) partagent la forme name, icon et options, où options est la liste des entrées.

EmoteData.Propsprops.lua{ name, icon, options }Émotes attachant un ou plusieurs objets au ped.
EmoteData.Dancesdances.lua{ name, icon, options }Danses solo.
EmoteData.SyncDancesgénéré au runtime{ name, icon, options }Version synchronisée des danses (voir plus bas).
EmoteData.Kitkit.lua{ name, icon, options }Émotes KIT natives du jeu (KIT_EMOTE_*).
EmoteData.Synchronizedsynchronized.lua{ name, icon, options }Émotes à deux joueurs (câlins, demande en mariage...).
EmoteData.Horsehorse.lua{ name, icon, options }Animations jouées sur le cheval le plus proche.

EmoteData.Walkswalks.lualiste plateStyles de marche MP_Style_*.

Structure d'une émote ​

Une entrée d'émote standard (catégories General, Props, Dances, Synchronized) suit ce schéma.

ChampTypeDescription
LabelstringNom affiché dans le menu.
CommandstringIdentifiant unique, sert de clé d'index et de commande chat.
Dictionarystring | string[]Dictionnaire d'animation. Si Dictionary et Animation sont des tableaux, une paire est tirée au hasard.
Animationstring | string[]Animation à jouer.
playableBystring[]Restreint l'émote (mp_male, mp_female, all).
isVipbooleanRéserve l'émote aux joueurs VIP (rang VipMinRank).
isAdminbooleanRéserve l'émote au staff (AdminLevel).
AdvancedtableJoue l'animation via TaskPlayAnimAdvanced avec Coords et Heading.
OptionstableDrapeaux, durée, props, émote secondaire, configuration de synchronisation.

Le sous-objet Options accepte les champs suivants.

ChampTypeDescription
Flags.LoopbooleanJoue l'animation en boucle.
Flags.MovebooleanAutorise le déplacement pendant l'émote.
Flags.StuckbooleanAjoute le flag « bloqué ».
DelaynumberAttente (ms) avant de lancer l'animation.
durationnumberDurée forcée (ms). Sinon calculée via GetAnimDuration, ou 2500 ms par défaut.
Propstable[]Objets attachés au ped (voir ci-dessous).
SecondaryEmotetableAnimation secondaire (Dictionary, Animation, Duration).
SharedtableParamètres d'une émote synchronisée.

Émotes avec objets (props) ​

Dans EmoteData.Props, chaque entrée d'Options.Props décrit un objet attaché.

ChampTypeDescription
NamestringModèle de l'objet (résolu via joaat).
BonenumberIndex d'os d'attache.
PlacementtableDeux vec3 : offset de position et offset de rotation.
VariationstableVariations de texture indexées par variation (optionnel).
lua
-- Extrait de EmoteData.Props
{
    Label = 'Torche',
    Command = 'torch',
    Animation = 'mainhand_idle',
    Dictionary = 'mech_doors@arms@torch',
    isVip = true,
    Options = {
        Flags = { Loop = true, Move = true },
        Props = {
            {
                Bone = 7966,
                Name = 's_interact_torch',
                Placement = {
                    vec3(0.04, 0.08, -0.02),
                    vec3(-91.0, 10.0, -10.0),
                },
            },
        },
    },
}

Émotes KIT ​

Les émotes KIT s'appuient sur les émotes natives du jeu plutôt que sur un dictionnaire d'animation.

ChampTypeDescription

Émotes synchronisées ​

Une émote Synchronized = true exige un partenaire à proximité. Son Options.Shared décrit le positionnement relatif et l'émote jouée par l'autre joueur.

ChampTypeDescription
OtherEmotestringCommand de l'émote jouée par le partenaire.
OtherAnimationstringAnimation alternative pour le partenaire (optionnel).
OtherDictionarystringDictionnaire alternatif pour le partenaire (optionnel).
FrontOffsetnumberDécalage avant/arrière.
SideOffsetnumberDécalage latéral.
HeightOffsetnumberDécalage vertical.
HeadingOffsetnumberDécalage d'orientation.
AttachbooleanAttache un joueur à l'autre au lieu de le repositionner.
Bone / Placementnumber / tableOs et offsets d'attache si Attach.

Les danses synchronisées (EmoteData.SyncDances) sont générées au démarrage à partir de EmoteData.Dances : la commande est préfixée par s et l'émote devient synchronisée avec elle-même (OtherEmote = sa propre commande).

Animations de cheval ​

Dans EmoteData.Horse, chaque entrée porte un Dictionary, une Animation et un Options avec Flag (flag d'animation natif) et Duration. L'animation est jouée sur le cheval monté ou le cheval le plus proche (rayon 5 unités).

Styles de marche, expressions, combat ​

lua
-- walks.lua  : { Label, Command, Walk = 'MP_Style_*' }
-- expressions.lua : { Label, Command, Expression = { idleName, dictionary } }
-- brawling.lua : { Label, Command, Hash = 0x........ }

Décals ​

EmoteData.Decals.options.damage contient name (catégorie) et decals, une liste de { name, label }. Chaque décal s'applique sur tout le corps via native.

Index (lookups) ​

Au chargement, entry.lua construit EmoteLookup pour permettre la résolution par commande. Les clés sont les Command en minuscules.

ChampContenu
EmoteLookup.emotesÉmotes de General, Props, Dances, Kit, Synchronized, SyncDances.
EmoteLookup.scenariosScénarios.
EmoteLookup.horseAnimations de cheval.

Jouer une émote ​

L'objet global Emote expose les fonctions de lecture.

Emote.PlayEmote ​

lua
Emote.PlayEmote(data, variation)

Joue une entrée d'émote complète (table issue de EmoteData). Gère la validation du ped (playableBy, ped humain), le chargement du dictionnaire, les props, les flags de mouvement, la durée, l'émote secondaire et l'arrêt automatique. Si data.Synchronized est vrai sans StartSynchronized, déclenche une requête vers le joueur le plus proche.

lua
local emote = EmoteLookup.emotes['dance']
if emote then
    Emote.PlayEmote(emote)
end

Emote.PlayByCommand ​

lua
Emote.PlayByCommand(command, variant)

Résout command (insensible à la casse) dans l'ordre scénarios → cheval → émotes, puis joue l'élément trouvé. Notifie une erreur si la commande est inconnue.

lua
Emote.PlayByCommand('drink')

Emote.PlayScenario ​

lua
Emote.PlayScenario(data)

Lance un scénario natif. data peut être une chaîne (nom du scénario) ou une table { Scenario, Command }. Impossible en véhicule ou à cheval.

Emote.PlayHorseByCommand / Emote.HorseAction ​

lua
Emote.PlayHorseByCommand(command)
Emote.HorseAction(action)
Emote.CancelHorse()

PlayHorseByCommand joue une animation sur le cheval le plus proche. HorseAction enchaîne une séquence prédéfinie (rear, scratch, dance, jump, rest, drink, roll, sleep, feet). CancelHorse interrompt l'animation du cheval.

Emote.Cancel ​

lua
Emote.Cancel(skipReset)

Arrête l'émote en cours : détache le ped, nettoie les tâches, supprime les props, réinitialise l'état et l'éventuelle synchronisation. skipReset conserve la dernière émote (utilisé pour la reprise après ouverture de porte).

Commandes ​

Les commandes chat sont enregistrées dans entry.lua.

CommandeArgumentEffet
/emotes, /emotemenu—Ouvre le menu.
/e[command] [variant]Joue l'émote/scénario/cheval ; sans argument ouvre le menu ; c annule.
/w[command]Applique un style de marche ; c réinitialise.
/f[command]Applique une expression faciale ; c réinitialise.
/brawl<command>Applique un style de combat.
/d[name]Bascule un décal de la catégorie damage ; c efface tous les décals.
/slowwalk<ratio>Définit le ratio de marche (borné [0.1, 3.0]).
lua
-- Exemples
/e dance
/e syncdance1     -- émote synchronisée : cible le joueur le plus proche
/w pistolero
/f colere
/brawl bruiser
/slowwalk 0.6

Synchronisation réseau ​

Les émotes synchronisées passent par le serveur (server/main.lua) qui valide l'accès, la distance et limite le débit.

ÉvénementSensDescription
lo:emotes:requestSyncclient → serveurDemande de synchronisation vers une cible. Le serveur vérifie l'accès et la distance (5 unités), puis relaie.
lo:emotes:syncRequestserveur → clientReçue par la cible : accepte avec Y, refuse avec N (auto-accepté si l'émetteur est staff).
lo:emotes:syncResponseclient → serveurRéponse d'acceptation, relayée aux deux joueurs.
lo:emotes:targetStartSyncserveur → clientDémarre l'émote côté cible.
lo:emotes:senderStartSyncserveur → clientDémarre l'émote côté émetteur, avec repositionnement.
lo:emotes:setCommandclient → serveurMet à jour l'état currentEmoteCommand du joueur.

WARNING

Les événements serveur sont déclarés via Net:SecureEvent avec maxPerSecond = 5. Les requêtes de synchronisation sont rejetées si les deux joueurs ne sont pas à portée (5 unités côté serveur) ou si l'accès (isAdmin / isVip) n'est pas satisfait.
