Module Inventaire ​

Le domaine inventory gère l'ensemble des inventaires du serveur : inventaire des joueurs, coffres (stash), objets au sol (drop), conteneurs, magasins (shop), tables d'artisanat et dons entre joueurs. C'est un domaine plat (inventory/server/*.lua, inventory/client/*.lua) chargé via le fxmanifest.lua — il n'y a pas de registre de modules. Tous les inventaires sont conservés dans un registre mémoire côté serveur et exposés via l'objet global Core.Inventory. Le domaine injecte des fonctions de commodité dans l'objet Player et s'accroche aux hooks de cycle de vie (Core.OnPlayerLoaded/Core.OnPlayerSave) ; le noyau ne dépend jamais de l'inventaire.

Cette page documente l'API serveur réelle, le format des items, les métadonnées d'armes, la configuration et les points d'extension (jo.hook).

Identifiants

Les noms d'items standards sont stockés en minuscules, les noms d'armes en majuscules (WEAPON_*). Core.Inventory.Items(name) résout indifféremment la casse.

Concept ​

Registre d'inventaires ​

Chaque inventaire est un objet placé dans un registre interne, indexé par son id. Pour un inventaire joueur, l'id est le source du joueur ; pour les autres types, c'est une chaîne (drop-1, gift:src, un identifiant de coffre, etc.).

On récupère un inventaire avec Core.Inventory.Get(id). La plupart des fonctions de l'API acceptent indifféremment un id ou directement un objet inventaire (utile pour chaîner sans relookup).

lua
local inv = Core.Inventory.Get(source)
print(inv.label, inv.weight, inv.maxWeight, inv.slots)

Types d'inventaires ​

TypeDescription
playerInventaire d'un joueur connecté. id = source. Persisté dans characters.inventory.
stashCoffre partagé ou personnel. Persisté dans la table inventories. Peut être restreint par groups, coords + distance.
dropObjets au sol, éphémères. Jamais persistés (datastore = true). Supprimés automatiquement quand vides et fermés.
containerInventaire interne à un item-conteneur (sac, sacoche). Créé à la volée depuis la métadonnée container du slot.
shopMagasin en lecture seule pour l'achat. Défini dans Core.InventoryData.Shops.
craftingTable d'artisanat avec recettes et dépôt d'ingrédients.
giftInventaire temporaire à 1 slot servant à donner un item à un joueur proche.

Format d'un item de slot ​

Un slot occupé contient une table avec les champs suivants :

ChampTypeDescription
namestringIdentifiant de l'item.
labelstringLibellé affiché.
descriptionstringDescription.
slotnumberIndex du slot.
countnumberQuantité dans le slot.
stackbooleanEmpilable.
closebooleanFerme l'inventaire à l'utilisation.
weightnumberPoids total du slot (item + métadonnées).
metadatatableMétadonnées (durabilité, munitions, conteneur, serial...).

Export GetInventory

L'export GetInventory normalise la sortie : chaque slot reçoit aussi amount (alias de count) et info (alias de metadata), et l'objet expose maxWeight et maxweight. Cet alias facilite l'interopérabilité ; en interne, utilisez toujours count et metadata.

Slots et poids ​

Le poids d'un slot est calculé par Core.Inventory.SlotWeight(item, slotItem) : poids de base (item.weight * count) auquel s'ajoutent le poids des munitions chargées (metadata.ammoType + metadata.ammo), le poids des composants (metadata.components) et un éventuel metadata.weight. Un item ne peut être ajouté que si le poids total reste sous maxWeight et qu'un slot est disponible (slot vide, ou slot d'empilage compatible).

API serveur ​

Toutes les fonctions ci-dessous vivent sous Core.Inventory. La majorité sont aussi exposées en exports (voir plus bas).

Ajout et retrait ​

lua
Core.Inventory.AddItem(id, name, count, metadata, slot)

Ajoute count exemplaires de name. Retourne true puis une table descriptive { name, count, slot, metadata } en cas de succès, ou false et un code d'erreur (invalid_inventory, invalid_item, invalid_count, inventory_full, too_heavy). Si l'item n'est pas empilable et count > 1, l'ajout est répété slot par slot. Le slot cible est optionnel : sinon, le premier slot d'empilage compatible ou le premier slot vide est choisi.

lua
-- Donner 2 pains à un joueur
local ok, resp = Core.Inventory.AddItem(source, 'bread', 2)

-- Ajouter une arme avec un serial imposé sur un slot précis
Core.Inventory.AddItem(source, 'WEAPON_REVOLVER_CATTLEMAN', 1, { serial = '123456ABC123456' }, 3)
lua
Core.Inventory.RemoveItem(id, name, count, metadata, slot)

Retire count exemplaires. Si metadata est fourni, seuls les slots dont les métadonnées correspondent (filtre partiel) sont concernés. Si slot est fourni, le retrait est limité à ce slot. Retourne true, ou false et un code (not_enough_items...).

lua
if Core.Inventory.RemoveItem(source, 'bandage', 1) then
    -- soin appliqué
end

Lecture et recherche ​

lua
Core.Inventory.GetItemBySlot(id, slot)       -- table du slot ou nil
Core.Inventory.GetSlotsWithItem(id, name, metadata) -- liste de slots
Core.Inventory.GetItemCount(id, name, metadata)     -- quantité totale
Core.Inventory.HasItem(id, items, amount)    -- boolean
Core.Inventory.Search(id, mode, names, metadata)

HasItem accepte une chaîne, une liste de noms, ou une table name = needed. Avec une liste ou une table, toutes les entrées doivent être satisfaites.

lua
-- Un seul item
if Core.Inventory.HasItem(source, 'lockpick') then ... end

-- Plusieurs items avec quantités requises
if Core.Inventory.HasItem(source, { lockpick = 1, screwdriver = 2 }) then ... end

Search fonctionne en deux modes : mode == 'count' renvoie un total numérique pour les noms demandés, sinon renvoie une table name = slots[].

lua
local total = Core.Inventory.Search(source, 'count', { 'money', 'cents' })
local slotsByName = Core.Inventory.Search(source, 'slots', 'bread')

Capacité (CanCarry) ​

lua
Core.Inventory.CanCarryWeight(id, weight)        -- poids brut
Core.Inventory.CanCarryItem(id, name, count, metadata)
Core.Inventory.CanCarryAmount(id, name)          -- nombre max transportable

CanCarryItem vérifie à la fois le poids et la disponibilité d'un slot. CanCarryAmount renvoie combien d'exemplaires de name tiennent dans le poids restant (math.huge pour un item sans poids).

lua
if not Core.Inventory.CanCarryItem(source, 'gold_bar', 5) then
    Core.Notify(source, 'Inventaire plein', 'error')
    return
end

Métadonnées et nettoyage ​

lua
Core.Inventory.SetMetadata(id, slot, metadata)   -- remplace la metadata d'un slot
Core.Inventory.Clear(id, keep)                   -- vide, en conservant keep
Core.Inventory.MetadataEquals(a, b)              -- comparaison profonde
Core.Inventory.CopyMetadata(metadata)            -- copie profonde

Clear accepte un keep optionnel : une chaîne (un nom à conserver) ou une liste de noms. Le reste est supprimé.

lua
-- Confisquer tout sauf l'argent
Core.Inventory.Clear(source, { 'money', 'cents' })

Items utilisables (CreateUseableItem) ​

lua
Core.Inventory.CreateUseableItem(name, callback)

Enregistre un callback serveur déclenché quand le joueur utilise l'item. Le callback reçoit (source, data) où data = { name, slot, metadata }. La consommation (champ consume) est appliquée avant l'appel du callback.

lua
Core.Inventory.CreateUseableItem('radio', function(source, data)
    -- ouvrir l'interface radio pour data.slot
end)

À chaque utilisation, le module déclenche également l'event serveur lo:Server:UsedItem avec (source, name, slot, metadata).

Persistance ​

lua
Core.Inventory.SaveInventory(id)   -- sauvegarde immédiate (sauf datastore)
Core.Inventory.SaveAll()           -- sauvegarde tous les inventaires modifiés

L'inventaire joueur est sérialisé en JSON minimal [{ name, count, slot, metadata? }] dans characters.inventory. Les autres types (sauf datastore) sont écrits dans la table inventories (owner, name, data, lastupdated). Un thread de sauvegarde périodique tourne toutes les Config.Inventory.saveInterval ms ; les coffres inactifs sont purgés au démarrage selon Config.Inventory.keepStashes.

Sauvegarde paresseuse

Un inventaire n'est écrit que si son flag interne changed est vrai. Les fonctions d'écriture (AddItem, RemoveItem, SetMetadata, Clear, swap) le positionnent automatiquement.

Référence des items ​

Les items sont single-source en base de données (table items), chargés en cache au boot par Core.Items (items/server/items.lua). Les fonctions ci-dessous lisent ce cache :

lua
Core.Inventory.Items(name)         -- définition d'un item (ou nil) -- alias de Core.Items:Get
Core.Inventory.GetItemList()       -- table complète des items -- alias de Core.Items:GetAll
Core.Inventory.RegisterItem(name, def) -- enregistre/met à jour un item à chaud -- alias de Core.Items:Upsert

Chaque domaine déclare inline les items dont il a besoin via Core.Items:Ensure(list) dans son Core.Ready : l'item est inséré en DB seulement s'il n'existe pas (jamais d'écrasement), puis mis en cache.

lua
Core.Ready(function()
    Core.Items:Ensure({
        { name = 'bread', label = 'Pain', weight = 250, description = 'Petit pain', useButton = 'Manger' },
    })
end)

Fonctions injectées dans l'objet Player ​

Au chargement du joueur, le module injecte des raccourcis dans player.Functions. Ils opèrent sur l'inventaire du joueur courant.

FonctionÉquivalent
player.Functions.AddItem(name, count, slot, metadata)Core.Inventory.AddItem
player.Functions.RemoveItem(name, count, slot)Core.Inventory.RemoveItem
player.Functions.HasItem(items, amount)Core.Inventory.HasItem
player.Functions.GetItemBySlot(slot)Core.Inventory.GetItemBySlot
player.Functions.GetItemByName(name)premier slot contenant l'item
player.Functions.GetItemsByName(name)tous les slots contenant l'item
player.Functions.ClearInventory(keep)Core.Inventory.Clear

Format de retour

GetItemBySlot, GetItemByName et GetItemsByName normalisent le slot : ajout de amount (alias de count) et info (alias de metadata).

lua
local player = Core.Players[source]
if player.Functions.HasItem('key') then
    player.Functions.RemoveItem('key', 1)
end

Définition d'un item ​

Les items génériques sont stockés en base (Core.Items, déclarés via Core.Items:Ensure), tandis que les armes statiques vivent dans Core.InventoryData.Weapons (inventory/shared/data/weapons.lua). Voici les champs reconnus dans une définition d'item.

Item standard ​

ChampTypeDescription
labelstringLibellé affiché (défaut : le nom).
weightnumberPoids unitaire (défaut 0).
stackbooleanEmpilable (défaut true).
closebooleanFerme l'inventaire à l'usage (défaut true).
descriptionstringDescription.
consumenumberQuantité consommée à l'usage. >= 1 retire des exemplaires ; entre 0 et 1, dégrade la durabilité.
degradenumberDurée (en minutes) avant péremption (durabilité temporelle).
durabilityboolean ou numberActive la durabilité. Déduit automatiquement si degrade ou un consume fractionnaire est présent.
ammoTypestringType de munition (active ammo = true).
useButtonstringLibellé du bouton d'usage. Déduit du label si absent (Boire, Manger, Lire, Fumer).
clienttableDonnées client : usetime, anim, prop, status, export.
servertableDonnées serveur : export.
containertable{ slots, maxWeight } : fait de l'item un conteneur (voir ci-dessous).
lua
Core.Items:Ensure({
    {
        name = 'bread',
        label = 'Pain',
        weight = 250,
        stack = true,
        close = true,
        description = 'Petit pain',
        client = { usetime = 2500 },
    },
})

Item-conteneur ​

Si la définition d'un item porte un champ container = { slots, maxWeight }, l'ajout d'un exemplaire génère une métadonnée container (identifiant unique) et size (slots + poids max). L'ouverture du conteneur crée un inventaire de type container à la volée.

lua
Core.Items:Ensure({
    { name = 'paperbag',  label = 'Sac en papier', weight = 50,  container = { slots = 5,  maxWeight = 1000 } },
    { name = 'backpack1', label = 'Sac à dos',      weight = 500, container = { slots = 40, maxWeight = 100000 } },
})

Armes et métadonnées ​

Une arme est un item de Core.InventoryData.Weapons. À l'enregistrement, le module force weapon = true, stack = false (sauf throwable), calcule hash = joaat(model), applique une durability par défaut de 0.05 et un attachPoint de 0.

ChampTypeDescription
labelstringLibellé.
weightnumberPoids.
modelstringModèle (défaut : le nom). Sert au calcul du hash.
durabilitynumberFacteur d'usure par tir/coup.
attachPointnumberPoint d'attache à la ceinture.
allowedAmmostableMunitions acceptées : { AMMO_X = true, ... }.
throwablebooleanArme de lancer (empilable).
lua
Core.InventoryData.Weapons.WEAPON_BOW = {
    label = 'Bow',
    weight = 1000,
    durability = 0.03,
    attachPoint = 13,
    allowedAmmos = {
        AMMO_ARROW = true,
        AMMO_ARROW_FIRE = true,
        AMMO_ARROW_POISON = true,
    },
}

Métadonnées d'arme ​

À la création (BuildMetadata), une arme reçoit ces métadonnées :

CléDescription
durabilityPourcentage de durabilité (défaut 100).
ammoMunitions chargées (défaut 0 si allowedAmmos).
ammoTypeType de munition actuellement chargée.
componentsListe des composants montés (défaut {} si allowedAmmos).
serialNuméro de série (6 chiffres + 3 lettres + 6 chiffres), généré sauf si registered == false.
registeredNom du propriétaire enregistré, ou false.
usedFlag de ré-équipement au respawn.

Durabilité temporelle

Un item standard avec degrade reçoit une durability égale à os.time() + degrade * 60 (timestamp d'expiration). Core.Inventory.UpdateDurability(inv, slot) met la durabilité à 0 une fois ce timestamp atteint. Une durabilité <= 100 est interprétée comme un pourcentage.

Composants ​

Un composant est un item dont la définition expose component = true, un type (catégorie d'emplacement, ex. tint, matériau) et une liste compatibleWeapons. Le montage (attachComponent) retire le composant de l'inventaire, remplace tout composant existant du même type et l'ajoute à metadata.components de l'arme. Le démontage (detachComponent) rend le composant à l'inventaire.

Grille personnalisée

Coffres, magasins et tables d'artisanat peuvent surcharger la grille via invCols, invRows et invMaxWeight. Core.Inventory.ResolveGrid(cfg) résout slots, cols, rows, maxWeight en repartant des valeurs de Config.Inventory par défaut.

Drops (objets au sol) ​

lua
Core.Inventory.CreateDrop(coords)         -- crée un drop à des coordonnées
Core.Inventory.RemoveDropIfEmpty(inv)     -- supprime le drop s'il est vide et fermé

Un drop a 10 slots, un poids max de 50000, et n'est jamais persisté. Le déplacement d'un item vers un nouveau drop se fait via le swap avec toType == 'newdrop' (le module crée le drop au pied du joueur).

lua
local ped = GetPlayerPed(source)
local drop = Core.Inventory.CreateDrop(GetEntityCoords(ped))
Core.Inventory.AddItem(drop, 'bread', 1)

Magasins (shops) ​

Les magasins sont enregistrés à chaud (par exemple depuis le panel lo_jobscreator), pas dans une table statique. Une config de shop porte un label, une liste items (name, price, currency) et éventuellement invCols/invRows/description.

lua
Core.Inventory.EnsureEditorShop(shopId, cfg, label)   -- enregistre un shop s'il n'existe pas encore
Core.Inventory.RegisterDynamicShop(shopId, cfg, label) -- (ré)enregistre un shop (écrase l'existant)
Core.Inventory.GetShopPayload(shopId)                  -- payload d'affichage

L'achat est géré par le callback lo:inventory:buyItem. La devise est normalisée : gold débite l'or, cents débite des centimes, tout le reste débite le cash. Un achat échoué pour cause d'inventaire plein rembourse automatiquement.

Commandes serveur ​

Ces commandes exigent le niveau de permission correspondant (Core.Permissions.HasCommandLevel). L'argument userId est l'identifiant unique du compte, résolu en source.

CommandeUsage
additem / giveitem/additem [userId] [item] [count] (ou un type d'argent).
removeitem/removeitem [userId] [item] [count].
givemoney`/givemoney [userId] [cash
setmoney`/setmoney [userId] [cash
showinv/showinv [userId] — affiche le contenu en console.

Points d'extension (hooks) ​

Le module expose des points d'extension via jo.hook. Les actions sont des notifications (doActions), les filtres peuvent annuler une opération (applyFilters).

HookTypeArguments
lo:inventory:createItemactioninvId, item, metadata, count
lo:inventory:usingItemactionsource, name, slot, consume
lo:inventory:openInventoryactionsource, invId, type
lo:inventory:openShopactionsource, shopId
lo:inventory:boughtItemactionsource, shopId, name, count, amount, moneyType
lo:inventory:swapItemsactionsource, fromId, toId, fromSlot, toSlot, count
lo:inventory:canSwapItemsfiltreallowed, source, fromId, toId, fromSlot, toSlot, count

Le filtre lo:inventory:canSwapItems doit retourner false pour bloquer un transfert. Il est notamment utilisé en interne par les inventaires de don et d'artisanat.

lua
jo.hook.registerFilter('lo:inventory:canSwapItems', function(allowed, src, fromId, toId, fromSlot, toSlot, count)
    if allowed == false then return allowed end
    -- empêcher toute sortie d'un coffre verrouillé
    if fromId == 'evidence_locker' then return false end
    return allowed
end)
