Roue de sélection (Wheel) ​

Le module wheel fait le pont entre l'inventaire de lo et la roue de sélection rapide native de RedM (hud_quick_select). Il maintient en permanence un miroir des objets consommables du joueur dans l'inventaire natif du jeu, de sorte que ces objets apparaissent dans la roue radiale. Lorsque le joueur sélectionne un objet dans la roue puis la referme, le module déclenche l'utilisation de l'objet correspondant côté inventaire lo.

L'objet global exposé est Core.Wheel. Le module est réparti sur trois fichiers (shared, client, server) qui se synchronisent via des events réseau.

INFO

La roue ne crée pas de « vrais » objets : elle injecte des entrées natives temporaires (gérées par le module) dont le nombre reflète la quantité réellement détenue dans l'inventaire lo. Ces entrées natives sont nettoyées au démarrage et à l'arrêt de la ressource.

Principe de fonctionnement ​

Le flux complet repose sur une synchronisation serveur vers client, puis sur un déclenchement client vers serveur lors de l'utilisation.

  1. Le serveur construit un instantané (snapshot) des quantités d'objets de roue détenues par le joueur, à partir de Core.Inventory.Get(src).
  2. Cet instantané est envoyé au client via lo:wheel:applySnapshot.
  3. Le client applique l'instantané sur l'inventaire natif : il ajoute ou retire des entrées natives pour que leur nombre corresponde aux quantités désirées.
  4. Quand le joueur ferme la roue sur un objet sélectionné (sur les sous-roues d'index 1 ou 2), le client envoie lo:wheel:use au serveur.
  5. Le serveur retrouve le slot d'inventaire lo correspondant et renvoie lo:wheel:useSlot au client, qui appelle Core.Inventory.UseSlot.

Synchronisation ​

Le serveur resynchronise les joueurs de deux manières :

  • Périodiquement : une boucle tourne toutes les syncPollMs millisecondes et n'envoie un snapshot que si la signature de l'inventaire a changé (comparaison via lastSignature).
  • À la demande : sur réception de lo:wheel:requestSync, ou en force après une utilisation (resyncAfterUseMs).

Côté client, au démarrage, le module attend startDelayMs puis demande une première synchronisation.

API partagée ​

Ces fonctions sont définies dans le fichier shared et donc disponibles côté client comme serveur.

FonctionRetourDescription
Core.Wheel.IsItem(name)booleanRenvoie true si name fait partie des objets de roue actifs (présent dans la liste interne, non exclu et déclaré dans Core.InventoryData.Items).
Core.Wheel.Names()tableRenvoie la liste triée des noms d'objets de roue actuellement actifs.
lua
if Core.Wheel.IsItem('consumable_peach') then
    -- l'objet est géré par la roue
end

for _, name in ipairs(Core.Wheel.Names()) do
    print(name)
end

TIP

La liste des objets de roue est figée dans le code du module (catalogue interne de consommables, nourriture, boissons, tabac, tonics, appâts...). Core.Wheel.Names() ne retient de ce catalogue que les objets qui ne sont pas dans Config.Wheel.exclude et qui existent réellement dans Core.InventoryData.Items. Un objet absent de la base d'objets lo n'apparaîtra jamais dans la roue.

Ajouter ou retirer des entrées ​

La liste des objets éligibles est interne au module ; il n'existe pas d'API publique pour l'étendre à chaud. Pour piloter ce qui apparaît dans la roue, deux leviers sont disponibles.

Inclure un objet ​

Un objet du catalogue interne apparaît automatiquement dans la roue dès lors qu'il est déclaré dans la base d'objets de l'inventaire (Core.InventoryData.Items). Le filtrage est appliqué à l'initialisation du fichier shared :

lua
for _, name in ipairs(WHEEL_ITEMS) do
    if not excludeSet[name] and (Core.InventoryData.Items or {})[name] then
        set[name] = true
        names[#names + 1] = name
    end
end

Concrètement : si l'objet existe dans les données d'inventaire et qu'il n'est pas exclu, il sera géré par la roue. Si vous ajoutez un consommable à votre base d'objets dont le nom figure dans le catalogue interne, il devient disponible sans configuration supplémentaire.

Exclure un objet ​

Pour retirer un objet de la roue sans le supprimer de l'inventaire, ajoutez son nom à Config.Wheel.exclude :

lua
Config.Wheel = {
    enable = true,
    exclude = {
        'consumable_cigar',
        'consumable_moonshine',
    },
    useCooldownMs = 350,
    syncPollMs = 2000,
    startDelayMs = 3000,
    resyncAfterUseMs = 900,
}

WARNING

Seul un nom présent dans le catalogue interne du module a un effet réel. Exclure un nom qui n'y figure pas est sans conséquence. De même, seul un objet natif valide (vérifiable par la commande wheelcheck) sera réellement affiché dans la roue du jeu.

Configuration ​

Le module lit la table Config.Wheel. Les valeurs ci-dessous sont celles par défaut.

CléTypeDéfautDescription
enablebooleantrueActive le module. À false, aucune des boucles client/serveur ne démarre. Côté client, la condition testée est WHEEL.enable ~= false.
excludetable{}Liste de noms d'objets à retirer de la roue, même s'ils existent dans la base d'objets.
useCooldownMsnumber350Délai minimum (ms) entre deux utilisations d'objet déclenchées par la roue, côté client. Empêche les déclenchements multiples rapprochés.
syncPollMsnumber2000Intervalle (ms) de la boucle serveur qui resynchronise les joueurs dont l'inventaire a changé.
startDelayMsnumber3000Délai (ms) attendu côté client avant la première demande de synchronisation au démarrage.
resyncAfterUseMsnumber900Délai (ms) après une utilisation avant que le serveur ne force une resynchronisation du joueur.

Events réseau ​

EventSensDonnéesDescription
lo:wheel:requestSyncclient vers serveuraucuneDemande au serveur un envoi forcé de l'instantané pour la source appelante.
lo:wheel:applySnapshotserveur vers clientsnapshotTable name -> quantité appliquée à l'inventaire natif du client.
lo:wheel:useclient vers serveurnameIndique que le joueur a utilisé l'objet name depuis la roue.
lo:wheel:useSlotserveur vers clientslotDemande au client d'utiliser le slot d'inventaire lo indiqué, via Core.Inventory.UseSlot.
lo:wheel:forceResyncclient (interne)aucuneDéclenche un lo:wheel:requestSync côté serveur.

INFO

Le serveur valide toujours la demande d'utilisation : si name n'est pas un objet de roue (Core.Wheel.IsItem) ou si aucun slot ne le contient (findSlot), il ignore l'utilisation et resynchronise le joueur pour corriger un éventuel décalage d'affichage.

Commande de diagnostic ​

Le client enregistre une commande pour vérifier quels objets de roue sont réellement reconnus par la base d'objets native du jeu.

bash
wheelcheck

Elle affiche dans la console deux listes : les objets natifs valides (visibles dans la roue) et les objets non natifs (invisibles dans la roue malgré leur présence dans la liste).

TIP

Utilisez wheelcheck lorsqu'un consommable n'apparaît pas dans la roue : s'il figure dans la liste des objets « NON natifs », c'est que sa clé n'est pas valide côté natif du jeu, et non un problème de configuration lo.

Cycle de vie ​

  • Au démarrage (onResourceStart implicite via le thread d'init) : le client construit son index hash -> nom, nettoie d'éventuelles entrées natives résiduelles, réinitialise son état miroir, attend startDelayMs, puis demande une synchronisation.
  • À l'arrêt (onResourceStop) : le client retire toutes les entrées natives gérées par la roue et réinitialise son état miroir, afin de ne pas laisser d'objets fantômes dans la roue native.
  • À la déconnexion : le serveur efface la signature mémorisée du joueur (Core.PlayerDropped).
