Éditeur de configuration (panels) ​

Éditeur in-game retiré

L'éditeur in-game /editor (architecture v1) est entièrement supprimé : plus de commande /editor, plus de table editor_overrides, plus d'auto-découverte de Config. La configuration staff se fait désormais via la ressource externe lo_jobscreator (système de panels).

config/*.lua reste la source unique des défauts. Les valeurs modifiées en jeu sont persistées en base (table lo_config) et appliquées par-dessus les défauts au boot. Chaque domaine de lo déclare un panel qui décrit ses champs, sait lire/écrire sa feuille Config et se publie dans lo_jobscreator.

Source

bridge/server/panelhost.lua (PanelHost, ConfigStore), bridge/server/typeconfighost.lua (TypeConfig), et les feuilles panels/server/*.lua.

Persistance — ConfigStore ​

Les overrides sont stockés dans la table lo_config, indexée par id (le data est un JSON).

lua
local data = ConfigStore.Load(id, migrateFile)
ConfigStore.Save(id, tbl)
  • ConfigStore.Load(id, migrateFile) lit la ligne lo_config pour id. Si la DB est vide et qu'un ancien fichier JSON (migrateFile) existe, il est importé une fois puis écrit en base. Renvoie nil si rien n'est trouvé.
  • ConfigStore.Save(id, tbl) sérialise tbl et l'upsert dans lo_config.

Déclarer un panel — PanelHost.Define ​

PanelHost.Define est la voie haut niveau : on fournit un schema (les champs) et un root (la feuille Config ciblée), et le panel se charge de tout — chargement des overrides, application sur Config, publication, sauvegarde.

lua
PanelHost.Define({
    id = 'survival',
    label = 'Survie & Gameplay',
    description = 'Faim/soif, régénération, PVP, santé et endurance.',
    icon = 'star',
    section = 'system',
    order = 94,
    schema = SCHEMA,
    file = 'data/survival_overrides.json',
    root = function() return Config end,
    publish = function() GlobalState.cfgSurvival = PanelHost.Collect(SCHEMA, Config) end,
    onSave = function() end,
})
ChampRôle
idIdentifiant unique (clé lo_config + clé du panel)
label / description / iconMétadonnées d'affichage dans lo_jobscreator
section / orderRegroupement et ordre dans l'UI (section défaut 'system')
schemaListe de champs (voir ci-dessous)
rootfun():table renvoyant la feuille Config à lire/écrire
fileAncien fichier JSON à migrer si la DB est vide (optionnel)
publishAppelée après chaque chargement/sauvegarde (ex. republier en GlobalState)
onSaveHook supplémentaire après application + persistance

Define enregistre le panel dans Ready : il appelle ConfigStore.Load, applique les valeurs sur root(), exécute publish, puis publie le panel. À la sauvegarde, les valeurs sont appliquées sur Config, persistées via ConfigStore.Save, et publish / onSave rejoués.

Charger la config dans Ready

Tout chargement de config (et tout accès DB) doit se faire dans Ready, jamais au top-level d'un fichier ni dans un CreateThread. Define s'en occupe déjà ; un panel bas niveau doit le faire lui-même.

Schéma de champs ​

Le schema est une liste de champs. Types courants : number (avec min/max/step), text, checkbox, list (tableau d'objets via itemFields), et submenu (regroupement de fields). La key est un chemin pointé relatif à root() (ex. Hud.Metabolism.interval).

lua
local SCHEMA = {
    { type = 'submenu', key = '_metab', label = 'Faim & soif', fields = {
        { type = 'number', key = 'Hud.Metabolism.interval', label = 'Intervalle (ms)', min = 5000, max = 600000, step = 1000 },
        { type = 'number', key = 'Hud.Metabolism.hungerRate', label = 'Perte de faim / tick', min = 0, max = 20, step = 0.1 },
    } },
}

PanelHost.Collect(schema, root) extrait les valeurs courantes du schéma, et PanelHost.Apply(schema, root, values) réapplique un jeu de valeurs (avec clamp min/max pour les nombres).

Enregistrement bas niveau — PanelHost.Register ​

Quand la config n'est pas une simple feuille Config (ex. données possédées par un service), on enregistre directement le panel avec ses propres get / save.

lua
PanelHost.Register({
    id = 'lottery',
    label = 'Loterie',
    icon = 'sparkles',
    section = 'system',
    order = 99,
    fields = {
        {
            key = 'prizes', type = 'list',
            label = 'Gains possibles',
            itemFields = {
                { key = 'weight', label = 'Poids', type = 'number', min = 0 },
                { key = 'amount', label = 'Gain ($)', type = 'number', min = 0 },
            },
        },
    },
    get = function()
        return Lottery and Lottery:GetPanelConfig() or { prizes = {} }
    end,
    save = function(values)
        return (Lottery and Lottery:SavePanelConfig(values)) and true or false
    end,
})

Exports lo_jobscreator ​

Deux dispatchers partagés possèdent les exports uniques par ressource, consommés par lo_jobscreator.

ExportSourceRôle
JcGetPanelConfig(id)panelhost.luaRenvoie les valeurs courantes du panel id (panel.get())
JcSavePanelConfig(id, values)panelhost.luaApplique et persiste les valeurs (panel.save(values))
JcPanelAction(id, action, value, src, row)panelhost.luaExécute une action déclarée par le panel (panel.action(...))
JcGetTypeConfig(typeId)typeconfighost.luaConfig globale d'un type d'interaction custom
JcSaveTypeConfig(typeId, values)typeconfighost.luaSauvegarde la config globale d'un type d'interaction

TypeConfig.Register(typeId, getFn, saveFn) branche un type d'interaction custom (déclaré côté lo_jobscreator) sur une config globale type-level, exposée via JcGetTypeConfig / JcSaveTypeConfig.

lua
TypeConfig.Register('butcher',
    function() return Hunting and Hunting:GetPrices() or {} end,
    function(values) return Hunting and Hunting:SavePrices(values) or false end
)

Republier au démarrage de lo_jobscreator ​

Si lo_jobscreator (re)démarre après lo, les panels déjà enregistrés sont automatiquement repoussés : panelhost.lua écoute onResourceStart et republie l'ensemble de PanelHost.panels. Aucune action manuelle n'est requise.
