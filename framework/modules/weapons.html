Armes ​

Domaine intégré à l'inventaire

La gestion des armes fait partie du domaine plat inventory (inventory/client/weapons.lua, inventory/server/weapons.lua), chargé via le fxmanifest.lua. Il n'y a pas de registre de modules ni de domaine weapons séparé. La logique vit sous Core.Inventory (notamment Core.Inventory.Weapon et Core.Inventory.Akimbo) et les données statiques sous Core.InventoryData.

Une arme est un item d'inventaire : elle occupe un slot, porte ses caractéristiques dans sa métadonnée (munitions, durabilité, composants, numéro de série) et s'équipe depuis la barre rapide. Cette page décrit le cycle équipement/munitions, l'akimbo, les composants et les points d'entrée serveur réels.

Données statiques ​

TableFichierContenu
Core.InventoryData.Weaponsinventory/shared/data/weapons.luaDéfinitions des armes (label, weight, model/hash, durability, attachPoint, allowedAmmos, throwable...)
Core.InventoryData.Componentsinventory/shared/data/components.luaComposants par type (Shared[type]) et par arme précise (Specific[weaponName])
Core.InventoryData.WeaponMapsConfig.Inventory.WeaponMapsTables de correspondance (TypeMap, Functional, Bows, Lanterns, ThrownAmmo...)
Core.InventoryData.AmmoSharedConfig.Inventory.AmmoMunitions : TypeHash, MaxAmmo, Labels, Types (groupes acceptés)

Définir une arme

Comme tout item, une arme est single-source en DB (Core.Items). Les définitions statiques de Core.InventoryData.Weapons servent de base ; voir Définition d'un item dans le module Inventaire pour les champs reconnus (weapon, durability, attachPoint, allowedAmmos, throwable).

Métadonnées d'arme ​

À la création, une arme reçoit dans sa métadonnée :

CléDescription
durabilityPourcentage de durabilité (défaut 100).
ammoMunitions chargées.
ammoTypeType de munition actuellement chargé.
componentsListe des composants montés.
serialNuméro de série généré, ou false si registered == false.
usedDrapeau de ré-équipement au respawn / reconnexion.

Équipement (client) ​

L'équipement est piloté depuis la barre rapide. Le client suit l'arme tenue en main via un planificateur (Core.Scheduler.Register('inventory:heldWeapon', ...)) et applique automatiquement les munitions et composants.

Fonction (Core.Inventory.Weapon)Rôle
Weapon.Equip(slotItem)Équipe l'arme du slot : résout le hash, charge les munitions, applique les composants, publie l'état
Weapon.Disarm(weapon)Range l'arme et nettoie l'état répliqué
Weapon.Give(p, hash, ammo, equipNow)Donne l'arme au ped et attend qu'elle soit présente
Weapon.LoadAmmo(p, hash, ammoTypeName, amount)Charge un type/quantité de munitions sur l'arme
Weapon.ApplyComponents(p, hash, weaponName, md)Applique les composants décrits dans la métadonnée

Quand le joueur appuie sur la touche d'usage d'un slot d'arme de la barre rapide, le client reçoit lo:inventory:useWeapon : il dégaine, range, ou bascule l'akimbo selon l'état courant.

Recharge ​

La touche R (Core.RegisterKey('reload_weapon', ...)) cherche le meilleur slot de munitions compatible et recharge l'arme tenue. La recharge est aussi déclenchable depuis l'interface via les callbacks NUI reloadAmmo / removeAmmo.

Munitions ​

Les munitions sont elles-mêmes des items (avec un champ ammoType). Charger des munitions retire l'item correspondant de l'inventaire et écrit le compte dans la métadonnée de l'arme.

Point d'entrée serveurTypeRôle
lo:inventory:updateWeaponcallback / event sécuriséMet à jour la métadonnée (load, ammo, throw) après tir/recharge/lancer
lo:inventory:loadInventoryAmmoevent sécuriséCharge des munitions depuis l'inventaire (respecte allowedAmmos et la taille de chargeur)
lo:inventory:removeAmmoFromWeaponcallbackDécharge l'arme et rend les munitions à l'inventaire
lo:inventory:mirrorAmmoevent sécuriséSynchronise le compte de munitions entre deux armes akimbo

Tous les events réseau entrants sont protégés par Core.SecureEvent (limitation de débit), et le serveur revalide chaque opération (l'arme existe, l'item munition est compatible, etc.).

Akimbo ​

Les armes une main pouvant être maniées en paire sont gérées par Core.Inventory.Akimbo (équipement gauche/droite, partage des munitions, ré-équipement). Principales fonctions :

Fonction (Core.Inventory.Akimbo)Rôle
Akimbo.IsAkimboGun(hash)Vrai si l'arme peut être maniée en paire
Akimbo.Equip(name, id) / Akimbo.Remove(name, id)Équipe / retire une arme akimbo
Akimbo.GetAkimboPartner(slot, hash)Renvoie le slot de l'arme jumelle
Akimbo.IsEquipped(id) / Akimbo.GetEquippedCount()État de la paire
Akimbo.Reset()Réinitialise le suivi (déconnexion / changement de perso)

Le serveur enregistre l'état d'usage de chaque slot via lo:inventory:setWeaponUsed et le slot actif via lo:inventory:setCurrentWeaponSlot.

Composants ​

Un composant est un item dont la définition expose component = true, un type (catégorie d'emplacement, ex. matériau, teinte) et une liste compatibleWeapons. Le montage retire le composant de l'inventaire, remplace tout composant existant du même type (rendu à l'inventaire) et l'ajoute à metadata.components de l'arme.

Callback serveurDonnéesRôle
lo:inventory:attachComponent{ componentSlot, weaponSlot }Monte un composant sur l'arme
lo:inventory:detachComponent{ weaponSlot, componentName }Démonte un composant et le rend à l'inventaire

Côté client (inventory/client/weapons.lua), les callbacks NUI attachComponent / detachComponent relaient ces actions, et l'application visuelle des composants suit un ordre de priorité (fonctionnel, matériau, gravure, matériau de gravure, teinte) via Core.InventoryData.Components.

Durabilité ​

La durabilité d'une arme (pourcentage dans metadata.durability) décroît à l'usage. Une arme à 0 est inutilisable (la recharge est refusée avec une notification). La logique de durabilité générique des items est décrite dans Définition d'un item du module Inventaire.
