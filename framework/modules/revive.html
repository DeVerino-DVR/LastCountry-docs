Réanimation / Médical ​

La réanimation et la mort des joueurs dans lo ne passent pas par un domaine revive ni par un global Revive : il n'existe plus de table Revive. La mécanique est répartie entre deux domaines :

  • admin/ : réanimation staff (commande /revive, net event lo:admin:reviveTarget) via une fonction locale et un signal de state bag.
  • death/ : machine à états de mort, réanimation par objet (kit de réanimation / médicament) et persistance de l'état entre les sessions.

Dans les deux cas, le serveur ne touche jamais directement à l'entité du joueur : il déclenche un net event ou bascule un state bag répliqué, et le client applique la résurrection native localement.

Pas de global Revive

La réanimation est implémentée par des fonctions locales (revive côté serveur, performRevive côté client) et des net events, dans admin/ et death/. Aucune table Revive n'est exposée comme global. Les wrappers globaux utilisés sont PlayerService, Players, Permissions, Notify, Logs, Commands, Net, Ped, Lifecycle et Items.

Réanimation staff (admin/) ​

Fonction locale revive ​

admin/server/revive.lua définit une fonction locale revive(src, userId). Elle résout la source réseau à partir du userId, marque l'état de mort comme résolu, puis incrémente le signal reviveSignal sur le state bag du joueur cible.

lua
local function revive(src, userId)
    userId = tonumber(userId)
    if not userId then return false end
    local targetSrc = PlayerService:SourceFromUserId(userId)
    if not targetSrc then
        if src ~= 0 then Notify(src, 'Joueur introuvable (userId).', 'error') end
        return false
    end
    local st = Player(targetSrc).state
    st:set('reviveSignal', (st.reviveSignal or 0) + 1, true)
    if src ~= 0 then Notify(src, ('Joueur #%s réanimé.'):format(userId), 'success') end
    if Logs then
        Logs.Send('admin', src, 'Réanimation', ('**Action :** `revive` (userId `#%s`)'):format(userId))
    end
    return true
end

Points importants :

  • La résolution userId vers source passe par PlayerService:SourceFromUserId. Si le joueur n'est pas connecté, la fonction renvoie false.
  • Le dernier argument true de state:set active la réplication du state bag vers le client concerné.
  • Si Logs est disponible, l'action est journalisée dans la catégorie admin.

Commande /revive ​

La commande staff /revive est enregistrée via Commands:Register. Elle accepte un userId optionnel ; sans argument, elle réanime le joueur de l'appelant.

lua
Commands:Register('revive', function(src, args)
    local target = args[1] and tonumber(args[1]) or nil
    if not target and src ~= 0 then
        local p = Players[src]
        target = p and p.userId or nil
    end
    if not target then
        if src ~= 0 then Notify(src, 'Joueur introuvable (userId).', 'error') end
        return
    end
    revive(src, target)
end, { level = 'revive', suggestion = 'Réanimer un joueur (userId, vide = soi-même)', suggestionParams = { { name = 'userId', help = 'ID unique (vide = soi-même)' } } })

Nomrevive

Niveau requis (level)revive

Résolution du niveau

L'option level = 'revive' est résolue par Commands:Register via Config.Perms.CommandLevels.revive, soit le niveau staff 20. Un appelant dont le niveau est inférieur reçoit un refus avant l'exécution du handler.

Net event lo:admin:reviveTarget ​

Un net event protégé permet à un panneau d'administration de demander une réanimation. Il est enregistré via Net:SecureEvent.

lua
Net:SecureEvent('lo:admin:reviveTarget', function(userId)
    local src = source
    if not Permissions:HasLevel(src, Config.Admin.Buttons.revive or 0) then return end
    revive(src, tonumber(userId))
end, { maxPerSecond = 5 })
Paramètre SecureEventValeurEffet
maxPerSecond5Plafond anti-flood par seconde.

Double contrôle de permission

Le niveau est vérifié explicitement dans le handler via Permissions:HasLevel. Un client qui ne possède pas le niveau requis ne peut donc pas déclencher de réanimation.

Application côté client ​

admin/client/revive.lua écoute le state bag reviveSignal sur son propre joueur et applique la résurrection native via la fonction locale performRevive.

lua
local function performRevive()
    local ped = Ped:PlayerPedId()
    if not ped or ped == 0 or not DoesEntityExist(ped) then return end

    local coords = GetEntityCoords(ped)
    local heading = GetEntityHeading(ped)

    if IsEntityDead(ped) or IsPedFatallyInjured(ped) then
        NetworkResurrectLocalPlayer(coords.x, coords.y, coords.z, heading, true, false, false)
    end

    SetEntityHealth(ped, GetEntityMaxHealth(ped))
    ClearPedTasksImmediately(ped)
    ClearPedBloodDamage(ped)
    Notify('Vous avez été réanimé par le staff.', 'success')
end

AddStateBagChangeHandler('reviveSignal', nil, function(bagName, _, value)
    if not value then return end
    if GetPlayerFromStateBagName(bagName) ~= PlayerId() then return end
    performRevive()
end)

Étapes appliquées :

ÉtapeNative / appelEffet
RésurrectionNetworkResurrectLocalPlayerRessuscite le joueur local à sa position et son cap actuels (uniquement si mort ou blessé à mort).
SantéSetEntityHealth(ped, GetEntityMaxHealth(ped))Remet la santé au maximum du ped.
TâchesClearPedTasksImmediatelyStoppe immédiatement toute animation ou tâche en cours.
SangClearPedBloodDamageNettoie les dégâts de sang visibles.

Wrapper ped

Le client utilise Ped:PlayerPedId() au lieu de la native brute, conformément à la convention du framework : le ped peut changer au respawn, ce wrapper fournit toujours la valeur à jour.

Réanimation par objet (death/) ​

Le domaine death gère la machine à états de mort et la réanimation entre joueurs via un objet. Un joueur portant un reviver_kit (ou un medicine) peut réanimer une cible proche.

lua
RegisterNetEvent('lo:Server:Death:RequestRevive', function(targetSrc)
    local src    = source
    local player = Players[src]
    local target = Players[targetSrc]
    if not player or not target then return end

    local hasReviver = player.Functions.HasItem('reviver_kit', 1)
    local hasMed     = not hasReviver and player.Functions.HasItem('medicine', 1)

    if not hasReviver and not hasMed then
        Notify(src, 'Vous n\'avez pas de quoi réanimer.', 'error')
        return
    end

    if hasReviver then
        player.Functions.RemoveItem('reviver_kit', 1)
    else
        player.Functions.RemoveItem('medicine', 1)
    end

    target:SetMetaData('deathState', S.ALIVE)
    TriggerClientEvent('lo:Client:Death:Revive', targetSrc, false)
end)

Les objets reviver_kit et medicine sont déclarés inline par le domaine dans son Ready via Items:Ensure.

EventSensRôle
lo:Server:Death:StateChangedclient → serveurPersiste le nouvel état (deathState) du joueur.
lo:Server:Death:RequestReviveclient → serveurRéanimation d'une cible : consomme un objet et bascule la cible en vie.
lo:Client:Death:Reviveserveur → clientDémarre la transition de réanimation sur la cible (byCommand distingue staff / objet).
lo:Client:Death:ForceStateserveur → clientForce l'état de mort du client (restauration à la connexion).

Persistance de l'état de mort ​

L'état de mort est persisté dans la métadonnée deathState du personnage, dont les valeurs proviennent de Enums.DeathState. À la connexion, le serveur lit cette métadonnée ; un personnage qui s'était déconnecté en état de mort, d'incapacité ou de respawn est ramené à ALIVE, et l'état est forcé sur le client via lo:Client:Death:ForceState.

lua
Lifecycle.OnPlayerLoaded(function(player)
    local state = player:GetMetaData('deathState') or S.ALIVE
    if state == S.DEAD or state == S.INCAPACITATED or state == S.RESPAWNING then
        state = S.ALIVE
        player:SetMetaData('deathState', S.ALIVE)
    end
    TriggerClientEvent('lo:Client:Death:ForceState', player.source, state)
end)

L'écriture de la métadonnée passe par player:SetMetaData(key, value), qui met à jour PlayerData.metadata et déclenche la mise à jour des données du joueur.

Respawn et coordonnées médicales ​

Le noyau de spawn de lo distingue plusieurs raisons de réapparition via Enums.SpawnReason :

lua
Enums.SpawnReason = {
    FIRST   = 'first',
    SWITCH  = 'switch',
    CREATOR = 'creator',
    RESPAWN = 'respawn',
    RESTART = 'restart',
}

La raison respawn correspond à une réapparition après la mort. Les coordonnées de ce respawn sont fournies par le domaine death via le hook de spawn du noyau, et non codées en dur dans le noyau lui-même.
