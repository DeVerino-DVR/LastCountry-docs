Module Réanimation / Médical ​

Le module revive gère la mort et la réanimation des joueurs dans lo. Il expose une API serveur pour réanimer un personnage par son charId, une commande staff /revive, un export, et un mécanisme de persistance de l'état de mort entre les sessions. Toute la mécanique de réanimation côté client repose sur la résurrection native du joueur local, déclenchée par des state bags répliqués depuis le serveur.

Le module est minimal et volontairement découplé : le serveur ne touche jamais directement à l'entité du joueur. Il bascule des jetons (tokens) dans le state bag du joueur, et le client réagit à ces changements pour appliquer la résurrection ou restaurer un état de mort.

Objet global

Tout passe par l'objet global Core. Le module enregistre sa table sous Core.Revive.

Vue d'ensemble ​

CôtéResponsabilité
ServeurMarque le personnage comme vivant (isdead, inlaststand), réplique un jeton de réanimation, journalise l'action, expose /revive et l'export RevivePlayer.
ClientÉcoute les state bags lo:revive et lo:dead, applique la résurrection native ou re-tue le ped au respawn pour restaurer une mort persistée.

Le couplage serveur vers client se fait uniquement via deux clés de state bag répliquées sur le joueur :

State bagSensRôle
lo:reviveserveur vers clientJeton incrémental ; chaque nouvelle valeur déclenche une réanimation locale.
lo:deadserveur vers clientJeton incrémental ; restaure l'état de mort à la connexion si le personnage était mort.

Côté serveur ​

Core.Revive.Player ​

Fonction centrale de réanimation. Elle résout la source réseau à partir du charId, remet les métadonnées du personnage à l'état vivant, puis répplique un jeton sur le state bag du joueur.

lua
---@param charId integer Identifiant unique du personnage à réanimer
---@param bySrc? integer Source du staff à l'origine de l'action (pour les logs)
---@return boolean ok, string? reason
local ok, reason = Core.Revive.Player(charId, bySrc)

Comportement détaillé :

lua
function Core.Revive.Player(charId, bySrc)
    charId = tonumber(charId)
    if not charId then return false, 'charId invalide' end

    local src = Core.Player.SourceFromCharId(charId)
    if not src then return false, 'joueur introuvable' end

    local player = Core.Players[src]
    if player and player.Functions and player.Functions.SetMetaData then
        player.Functions.SetMetaData('isdead', false)
        player.Functions.SetMetaData('inlaststand', false)
    end

    reviveToken = reviveToken + 1
    Player(src).state:set('lo:revive', reviveToken, true)

    if Core.Logs then
        Core.Logs.Send('admin', bySrc or 0, 'Réanimation', ('**Action :** `revive` (charId `#%s`)'):format(charId))
    end
    return true
end

Points importants :

  • La résolution charId vers source passe par Core.Player.SourceFromCharId. Si le joueur n'est pas connecté, la fonction renvoie false, 'joueur introuvable'.
  • Les métadonnées isdead et inlaststand sont remises à false via player.Functions.SetMetaData, ce qui déclenche la sauvegarde des données du personnage.
  • Le dernier argument true de state:set active la réplication du state bag vers le client concerné.
  • Si Core.Logs est disponible, l'action est journalisée dans la catégorie admin.

Valeurs de retour possibles :

okreasonCause
falsecharId invalideL'argument n'est pas convertible en nombre.
falsejoueur introuvableAucune source réseau ne correspond au charId.
truenilRéanimation déclenchée avec succès.

Export RevivePlayer ​

La fonction est aussi exposée comme export pour les ressources externes :

lua
exports('RevivePlayer', function(charId, bySrc) return Core.Revive.Player(charId, bySrc) end)

Exemple d'appel depuis une autre ressource :

lua
local ok, reason = exports.lo:RevivePlayer(charId, source)
if not ok then
    print(('Revive échoué : %s'):format(reason))
end

Commande /revive ​

La commande staff /revive est enregistrée via Core.RegisterCommand. Elle accepte un charId optionnel ; sans argument, elle réanime le personnage de l'appelant.

lua
Core.RegisterCommand('revive', function(src, args)
    local target = args[1] and tonumber(args[1]) or nil
    if not target then
        if src == 0 then
            print('[revive] usage : revive <charId>')
            return
        end
        local p = Core.Players[src]
        target = p and p.charId or nil
    end
    -- ...
    local ok, err = Core.Revive.Player(target, src)
end, { level = 'revive', suggestion = 'Réanimer un joueur (charId)', suggestionParams = { { name = 'charId', help = 'ID unique (vide = soi-même)' } } })
ÉlémentValeur
Nomrevive
ArgumentcharId (optionnel ; vide = soi-même)
Niveau requis (level)revive
Suggestion chatRéanimer un joueur (charId)

Résolution du niveau

L'option level = 'revive' est résolue par Core.RegisterCommand via Config.Perms.CommandLevels.revive, soit le niveau staff 20. Un appelant dont le niveau est inférieur reçoit un refus avant l'exécution du handler.

Comportement console (src == 0) :

  • Sans argument : affiche [revive] usage : revive <charId> et s'arrête.
  • En cas d'échec : affiche [revive] échec : <raison> dans la console serveur.

Comportement joueur :

  • Réussite : notification success du type Joueur #<charId> réanimé..
  • Échec : notification error du type Revive échoué : <raison>.

Event sécurisé lo:admin:reviveTarget ​

Un net event protégé permet à un panneau d'administration de demander une réanimation. Il est enregistré via Core.SecureEvent une fois ce dernier disponible.

lua
Core.SecureEvent('lo:admin:reviveTarget', function(charId)
    local src = source
    if not Core.Permissions.HasCommandLevel(src, 'revive') then return end
    Core.Revive.Player(charId, src)
end, { maxPerSecond = 5, minLevel = Config.Perms.CommandLevels.revive })
Paramètre SecureEventValeurEffet
maxPerSecond5Plafond anti-flood par seconde.
minLevelConfig.Perms.CommandLevels.revive (20)Niveau staff minimum requis ; en dessous, le handler n'est pas appelé.

Double contrôle de permission

Le niveau est vérifié à la fois par Core.SecureEvent (minLevel) et explicitement dans le handler via Core.Permissions.HasCommandLevel(src, 'revive'). Un client qui ne possède pas le niveau revive ne peut donc pas déclencher de réanimation.

Persistance de l'état de mort ​

À la connexion d'un joueur, le module vérifie si le personnage était mort lors de la dernière sauvegarde. Si c'est le cas, il répplique un jeton lo:dead pour que le client restaure l'état de mort.

lua
Core.OnPlayerLoaded(function(src)
    local player = Core.Players[src]
    local md = player and player.PlayerData and player.PlayerData.metadata
    if md and md.isdead then
        deadToken = deadToken + 1
        Player(src).state:set('lo:dead', deadToken, true)
    end
end)

Le drapeau lu est la métadonnée isdead du personnage. Le jeton deadToken est incrémental et répliqué, ce qui garantit qu'un changement de valeur est toujours détecté côté client.

Côté client ​

Réanimation locale ​

La réanimation locale est encapsulée dans doRevive, exposée publiquement sous Core.Revive.Local.

lua
local function doRevive()
    local ped <const> = Core.Ped.PlayerPedId()
    local playerId <const> = Core.Ped.PlayerId()
    local coords <const> = GetEntityCoords(ped)
    local heading <const> = GetEntityHeading(ped)

    NetworkResurrectLocalPlayer(coords.x, coords.y, coords.z, heading, 1, true, false)
    SetEntityHealth(ped, GetEntityMaxHealth(ped))
    ClearPedTasksImmediately(ped)
    SetPlayerInvincible(playerId, false)
    ClearPlayerWantedLevel(playerId)

    Core.Notify('Vous avez été réanimé.', 'success')
end
Core.Revive.Local = doRevive

Étapes appliquées :

ÉtapeNative / appelEffet
RésurrectionNetworkResurrectLocalPlayerRessuscite le joueur local à sa position et son cap actuels.
SantéSetEntityHealth(ped, GetEntityMaxHealth(ped))Remet la santé au maximum du ped.
TâchesClearPedTasksImmediatelyStoppe immédiatement toute animation ou tâche en cours.
InvincibilitéSetPlayerInvincible(playerId, false)Désactive l'invincibilité héritée de la mort.
RechercheClearPlayerWantedLevelRéinitialise le niveau de recherche.
RetourCore.NotifyAffiche une notification success.

Wrappers ped

Le module utilise Core.Ped.PlayerPedId() et Core.Ped.PlayerId() au lieu des natives brutes, conformément à la convention du framework : le ped peut changer au respawn, ces wrappers fournissent toujours la valeur à jour.

Core.Revive.Local peut être appelée directement côté client pour forcer une réanimation locale sans passer par le serveur :

lua
Core.Revive.Local()

Handler du state bag lo:revive ​

La réanimation est normalement déclenchée par la réplication serveur. Le client écoute le state bag lo:revive sur son propre joueur et déduplique via le dernier jeton reçu.

lua
local lastToken = nil

AddStateBagChangeHandler('lo:revive', nil, function(bagName, _, value)
    if value == nil then return end
    if bagName ~= ('player:%s'):format(GetPlayerServerId(PlayerId())) then return end
    if value == lastToken then return end
    lastToken = value
    doRevive()
end)

Logique de garde :

  • La valeur nil est ignorée.
  • Le handler ne traite que le state bag du joueur local (player:<serverId>).
  • Le jeton est comparé à lastToken pour éviter de réanimer deux fois sur la même valeur.

Handler du state bag lo:dead ​

À la connexion, si le serveur a répliqué lo:dead, le client attend que le joueur soit spawné puis remet sa santé à 0 pour restaurer l'état de mort persisté.

lua
local lastDead = nil

AddStateBagChangeHandler('lo:dead', nil, function(bagName, _, value)
    if value == nil then return end
    if bagName ~= ('player:%s'):format(GetPlayerServerId(PlayerId())) then return end
    if value == lastDead then return end
    lastDead = value
    CreateThread(function()
        local deadline <const> = GetGameTimer() + 20000
        while LocalPlayer.state.PlayerSpawned ~= true and GetGameTimer() < deadline do
            Wait(200)
        end
        Wait(500)
        local ped <const> = Core.Ped.PlayerPedId()
        if ped and ped ~= 0 and DoesEntityExist(ped) then
            SetEntityHealth(ped, 0)
        end
    end)
end)

Comportement :

  • Même garde de déduplication que lo:revive (lastDead).
  • Le thread attend l'état LocalPlayer.state.PlayerSpawned == true, avec un délai maximum de 20 secondes (deadline).
  • Après un délai de sécurité de 500 ms, la santé du ped est mise à 0, ce qui replace le joueur en état de mort dès son apparition.

Restauration de mort, pas réanimation

Le state bag lo:dead ne réanime pas : il sert à rejouer la mort d'un personnage qui s'était déconnecté en étant mort, afin que l'état médical reste cohérent entre les sessions.

Respawn et coordonnées médicales ​

Le noyau de spawn de lo distingue plusieurs raisons de réapparition via Core.Enums.SpawnReason :

lua
Core.Enums.SpawnReason = {
    FIRST   = 'first',
    SWITCH  = 'switch',
    CREATOR = 'creator',
    RESPAWN = 'respawn',
    RESTART = 'restart',
}

La raison respawn correspond à une réapparition après la mort. Les coordonnées de ce respawn sont fournies par le module médical via le hook de spawn du noyau, et non codées en dur dans le noyau lui-même. Le module revive documenté ici prend en charge la mort, la réanimation et la persistance de l'état ; la redirection des coordonnées de respawn est assurée par le point d'extension de spawn du framework.

Métadonnées du personnage ​

Le module lit et écrit les métadonnées suivantes sur le personnage :

MétadonnéeTypeRôle
isdeadbooleanIndique si le personnage est mort. Remis à false par Core.Revive.Player ; lu à la connexion pour décider de répliquer lo:dead.
inlaststandbooleanIndique l'état d'agonie (last stand). Remis à false par Core.Revive.Player.

L'écriture passe par player.Functions.SetMetaData(key, value), qui met à jour PlayerData.metadata, émet lo:Server:SetMetaData et déclenche la mise à jour des données du joueur.
