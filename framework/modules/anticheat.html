Anticheat ​

Le module Anticheat de lo est un système serveur-autoritaire de détection et de sanction. Il combine des heuristiques côté client (hooks de natives, scans périodiques) avec des vérifications côté serveur qui font foi : aucune décision de sanction n'est prise sur la seule base d'une remontée client. L'objet d'API est Anticheat, la configuration vit dans Config.Anticheat, et l'état effectif est répliqué aux clients via GlobalState.Anticheat.

Identifiants en anglais, explications en français

Tous les events, clés de config et exports restent en anglais (lo:anticheat:*). Seules les explications sont traduites.

Philosophie serveur-autoritaire ​

Le client détecte et signale, le serveur décide et sanctionne. Les principes structurants :

  • Les remontées client (lo:anticheat:report, heartbeat, sweeps) sont rate-limitées et ne déclenchent jamais directement un kick/ban : elles passent toujours par Anticheat.Sanction, qui applique l'action configurée.
  • Les détections lourdes (dégâts, kill distance, flood d'entités, mass kill, escalade de rôle, contournement de ban) sont entièrement calculées serveur à partir d'events de jeu (weaponDamageEvent, explosionEvent, entityCreating...) ou de la base de données.
  • Le client ne fait que faciliter : il pose des « grâces » au spawn, hooke les natives légitimes (GiveWeaponToPed, CreateCam, FreezeEntityPosition...) pour ne pas se signaler lui-même, et exécute les ordres serveur (strip d'armes, suppression d'entités).
  • Tout joueur de niveau staff suffisant (exemptLevel) est exempté ; cette exemption est répliquée dans Player(src).state.acExempt et lue côté client.
lua
-- L'exemption staff court-circuite toute sanction
function Anticheat.IsExempt(src)
    if not src or src == 0 then return true end
    if Anticheat.forceWatched[src] then return false end -- TestKit : surveillance forcée
    return Permissions:HasLevel(src, Anticheat.Config().exemptLevel)
end

Action globale et mode test ​

Trois réglages de premier niveau gouvernent toute la chaîne de sanction.

CléValeursEffet
Config.Anticheat.enabledtrue / falseCoupe l'anticheat entièrement (serveur et client).
Config.Anticheat.action'log' · 'kick' · 'ban'Sanction par défaut. 'log' n'enregistre que la détection.
Config.Anticheat.testModetrue / falseMode observation : tout est journalisé et enregistré en base, mais aucun kick ni ban n'est appliqué.

La logique de Anticheat.Sanction :

lua
-- Extrait de sanction.lua (logique effective)
local function logOnly()
    return isTestMode() or Anticheat.Config().action == 'log'
end
-- ...
recordDetection(src, reason, detail)   -- INSERT dans ac_detections
sendWebhook(src, reason, fields)       -- Logs.Send('anticheat', ...)
TriggerEvent('lo:anticheat:detection', src, name, reason)

if logOnly() then return end           -- testMode ou action 'log' : on s'arrête là

if forceBan or effective.action == 'ban' then
    Anticheat.Ban(src, reason)            -- UPDATE users SET banned = 1
    Anticheat.Drop(src, effective.banMessage, 'ban')
elseif effective.action == 'kick' then
    Anticheat.Drop(src, effective.kickMessage, 'kick')
end

testMode

En testMode, même Anticheat.Drop est neutralisé (il log au lieu de DropPlayer). Certaines détections passent un cinquième argument forceBan = true à Sanction (armes blacklist, spawn de model blacklisté, événements VORP interdits, escalade de rôle, injection de ressource...) : ces détections bannissent quel que soit action, sauf en testMode / action = 'log'.

Chaque détection est dédupliquée par un cooldown de log de 30 s (par couple src + checkKey) et persistée :

lua
-- Table de persistance des détections
INSERT INTO `ac_detections` (`timestamp`, `src_name`, `license`, `check_name`, `detail`) VALUES (?, ?, ?, ?, ?)

Système d'armes (sweep, strip, équipement) ​

Règle centrale : une arme non présente dans l'inventaire lo doit être retirée, pas seulement au tir.

Côté serveur (guard_weapons.lua), Anticheat.PlayerHasWeapon(src, weapon) compare le hash d'arme (normalisé via Anticheat.U32) au contenu de Inventory.Get(src). Trois déclencheurs alimentent la vérification :

DéclencheurSourceRéaction
weaponDamageEventnatif RDR3 (au tir)strip + sanction si l'arme n'est pas possédée
lo:anticheat:weaponEquippedclient (poll d'équipement)idem, cooldown equipCooldown
lo:anticheat:weaponSweepclient (scan complet)strip en batch des armes hors inventaire

Le client (weapons.lua) énumère toutes les armes du ped via HasPedGotWeapon sur la liste partagée Anticheat.Data.knownWeapons (récupérée par le callback lo:anticheat:getWeaponList, qui fusionne aussi les items-armes de l'inventaire). Les armes suspectes sont envoyées au serveur, qui répond avec lo:anticheat:stripWeapon (une arme) ou lo:anticheat:stripWeapons (batch). Les natives GiveWeaponToPed, RemoveWeaponFromPed, RemoveAllPedWeapons sont wrappées pour tracer les armes légitimes (legitWeapons) et éviter les faux positifs.

Les armes utilitaires sont ignorées (Anticheat.Data.weaponIgnore) : WEAPON_UNARMED, jumelles, appareil photo, détecteur de métaux, canne à pêche.

Mods d'armes

Config.Anticheat.weapons.mods (client weapon_mods.lua) surveille les altérations de dégâts / chargeur / munitions au-delà des tolérances (damageTolerance, clipTolerance, ammoTolerance) et peut réinitialiser (resetDamage, resetClip).

Config principale :

lua
weapons = {
    enabled = true,
    checkInventory = true,
    checkCooldown = 5000,    -- anti-spam de la vérif au tir
    equipCooldown = 10000,
    equipPoll = 500,
    spawnGrace = 20000,      -- délai de grâce après spawn/respawn
    client = {
        enabled = true,
        unownedDelay = 1000,
        sweepInterval = 750,
        inHandPoll = 500,
        reportCooldown = 3000,
    },
}

Garde d'entités (flood, watchdog, sweep) ​

Trois couches protègent contre le spam d'entités et les props de crash.

1. Garde à la création — guard_entity.lua écoute entityCreating :

  • Modèles de crash (Anticheat.Data.crashModels) : purge immédiate + sanction si flood (entity.crashFlood).
  • Modèles blacklistés (blacklistModels, animaux/armes lourdes/PNJ) : purge + sanction forceBan.
  • Incohérence d'archetype (entité à la fois objet et ped) : purge.
  • Flood par propriétaire (entity.flood.max sur entity.flood.window) : purge + nettoyage groupé via cleanupOwnerFlood.
  • deleteUnprotected (opt-in) : toute entité script-spawned non taguée lo_protected est supprimée après graceMs.

2. Watchdog périodique — watchdog.lua : à intervalle régulier, scanne toutes les entités par propriétaire (Anticheat.WatchdogScan), ignore les entités lo_protected, et supprime celles d'un propriétaire dépassant deleteThreshold. Un critère d'isolement (isolation × deuxième plus gros possesseur) distingue un cheateur d'une charge normale. Le watchdog surveille aussi le model swap : un ped joueur dont le model n'est pas dans modelSwap.playerModels (par défaut mp_male / mp_female) est sanctionné après modelSwap.strikes.

3. Sweep client — sweep.lua : pour les spawns locaux non-networked (invisibles du serveur), un scan client supprime les entités script-spawned (popType 0/7/8) non taguées lo_protected, en épargnant la population du jeu. Même garde deleteUnprotected.

La suppression réseau passe par Anticheat.SendDeleteNetIds(target, netIds), qui envoie lo:anticheat:deleteNetIds en chunks de 200 (le client borne à 256) pour ne jamais dépasser le plafond ~16 Ko des events fiables. Le client (watchdog.lua) demande le contrôle de l'entité (NetworkRequestControlOfEntity + boucle d'attente) avant DeleteEntity — un DeleteEntity serveur sur une entité possédée par un client ne ferait que la cacher.

lua
entity = {
    enabled = true,
    checkPeds = true, checkObjects = true, checkVehicles = true,
    scriptPopulationTypes = { [0] = true, [7] = true, [8] = true },
    archetypeMismatch = true,
    flood = { enabled = true, window = 10000, max = 25, cleanupCooldown = 3000 },
    crashFlood = { max = 3, window = 5000 },
    deleteUnprotected = { enabled = true, vehicles = true, objects = true, peds = true, graceMs = 1500 },
}

Dégâts, kill distance, explosions, mass kill ​

Dégâts — guard_damage.lua (sur weaponDamageEvent) :

ContrôleConditionRéaction
Arme blacklistdamage.blacklistWeapons[weapon]CancelEvent + sanction
Cause blacklistéeData.deathCauses[weapon] (ex. WEAPON_EXPLOSION)CancelEvent + sanction forceBan
Multi-victimes#hitGlobalIds >= damage.maxVictimsCancelEvent + sanction
Dégâts anormauxdmg > damage.maxDamageCancelEvent + sanction
Flood de dégâts> damage.max sur damage.windowCancelEvent + sanction

Kill distance : pour chaque victime joueur touchée, le serveur mesure la distance tireur↔victime. Au-delà de killDistance.default (ou perWeapon[hash]), après killDistance.strikes sur killDistance.window, sanction « impact longue distance / aimbot ».

Explosions & projectiles — guard_combat.lua : comptage par fenêtre (explosion.window), sanction au-delà de explosion.max. La whitelist Data.explosionWhitelist (types 24 lanterne, 31 flèche-dynamite) est exemptée.

Mass kill : sur lo:anticheat:death / lo:Server:OnPlayerDeath, plus de masskill.deaths morts imputées à un même tueur sur masskill.window déclenche une sanction. Le client (death.lua) détecte sa propre mort via CEventNetworkEntityDamage et remonte le killerServerId.

lua
damage = { enabled = true, maxDamage = 350, window = 10000, max = 60, maxVictims = 8,
           blacklistWeapons = { [2416459067] = true } },
killDistance = { enabled = true, strikes = 3, window = 30000, default = 500.0, perWeapon = {} },
explosion = { enabled = true, window = 10000, max = 10 },
masskill = { enabled = true, deaths = 5, window = 5000 },

Contrôles d'état (state checks) ​

Deux familles : des scans client (detect.lua, detect_extra.lua) qui remontent via lo:anticheat:report, et des scans serveur (state.lua).

Côté client, chaque contrôle accumule des strikes avant de signaler (Config.Anticheat.strikes par défaut). Les contrôles activables via Config.Anticheat.checks :

CheckDétecteAnti-faux-positif

Contrôles supplémentaires (detect_extra.lua, branche Config.Anticheat.extra) :

  • stamina : endurance qui ne descend pas en sprint prolongé (god-stamina).
  • freecam : distance caméra↔ped au-delà de maxDist avec confirmation, hors caméra script légitime.
  • noclip : vitesse réelle incohérente avec la vélocité physique, ou traversée de géométrie (shape-test).

Côté serveur (state.lua, intervalle state.interval) : santé > state.maxHealth, téléportation (saut de position > teleport.maxDist après teleport.strikes, alimenté par les coords du heartbeat), invisibilité confirmée serveur, et skin blacklisté. La téléportation est réinitialisée aux transitions de session (lo:session:stateChanged) et à la mort.

lua
state = {
    enabled = true, interval = 5000, maxHealth = 2100,
    teleport = { enabled = true, maxDist = 50.0, strikes = 3 },
    invisible = { enabled = true, strikes = 3 },
    skin = { enabled = false },
}

Intégrité (résolution 16:9, ressources injectées, event leurre) ​

Le module intégrité (integrity.lua) traite trois cas.

Résolution forcée : le client (integrity.lua client) calcule le ratio largeur / hauteur. Sous minRatio (1.6), il prévient le joueur, envoie lo:anticheat:integrity:resolution 'alert', laisse kickDelay secondes pour corriger, puis envoie 'kick'. Le serveur tient un filet : si le client devient muet après une alerte, un kick serveur est déclenché (cas du client patché qui n'honore pas l'auto-kick).

Injection de ressource : à chaque onClientResourceStart, le client remonte lo:anticheat:integrity:resource. Le serveur compare à une whitelist des ressources réellement démarrées (resourceWhitelist) et sanctionne (forceBan) toute ressource inconnue.

Event leurre : un event piège (decoyEventName, par défaut lo:anticheat:exploit) est enregistré côté serveur. Aucune partie légitime de lo ne le déclenche ; tout joueur qui le déclenche est sanctionné forceBan.

lua
integrity = {
    enabled = true, minRatio = 1.6, checkInterval = 1000, kickDelay = 30,
    reportResources = true, bootGrace = 30000, resourceWhitelist = true,
    decoyEvent = true, decoyEventName = 'lo:anticheat:exploit',
    kickMessage = 'Résolution non autorisée (écran étiré). Repassez en 16:9.',
}

Anti-évasion de ban (banEvasion) ​

À la connexion (Anticheat.ConnectCheck → BanEvasionCheck), le serveur compare les identifiants du joueur (license, license2, steam, discord, xbl, live, fivem, ip, tokens) à la table ac_ban_identity. Une correspondance d'identifiant unique, ou minTokenMatches tokens, refuse la connexion si refuseOnMatch. En testMode, la connexion est autorisée mais journalisée.

L'empreinte est capturée automatiquement lors d'un ban (Anticheat.CaptureBanIdentity, appelée par Anticheat.Ban) :

sql
SELECT `user_id`, `kind`, `value` FROM `ac_ban_identity`
-- capture au ban :
INSERT INTO `ac_ban_identity` (`user_id`, `kind`, `value`, `name`, `reason`, `timestamp`) ...

Les types capturés sont réglables via banEvasion.captureKinds / ignoreKinds.

RoleGuard (escalade de privilèges staff) ​

guard_roles.lua détecte une modification non sollicitée de admin_level en base. Toutes les roleGuard.interval ms, il compare le niveau attendu (event interne lo:perms:levelChanged ou valeur de session) au admin_level réel de la table users. Une divergence confirmée sur deux passes (anti-bruit) déclenche un log staff et, si sanctionOnSelfElevation et que le niveau a augmenté, une sanction forceBan pour escalade de privilèges.

lua
roleGuard = { enabled = true, interval = 30000, startupDelay = 20000, sanctionOnSelfElevation = true }

Garde de flood et events sécurisés ​

Flood d'events : deux hooks symétriques.

  • Entrant (guard_flood.lua, Anticheat.FloodHook.inbound) : au-delà de flood.max events / flood.window, les events sont droppés ; au-delà de flood.max × flood.sanctionFactor, sanction.
  • Sortant (flood.lua client, Anticheat.FloodHook.outbound) : throttle local à flood.client.perEvent / fenêtre.

Une whitelist exempte les events internes (lo:anticheat:*, lo:inventory:*, __cfx_*...). Ajouter un event à la whitelist :

lua
Anticheat.FloodWhitelist('mon:event:legitime')      -- entrant (serveur)
Anticheat.OutboundWhitelist('mon:event:legitime')   -- sortant (client)

Events sécurisés : Net:SecureEvent(name, handler, opts) enregistre un event réseau avec rate-limit, plafond d'octets et niveau de permission optionnel :

lua
Net:SecureEvent('lo:monmodule:action', function(payload)
    local src = source
    -- handler exécuté seulement si toutes les gardes passent
end, {
    maxPerSecond = 20,   -- défaut 20
    maxBytes = 8192,     -- défaut Config.Anticheat.maxEventBytes
    minLevel = nil,      -- niveau staff requis (sinon sanction)
    netOnly = false,     -- force RegisterNetOnlyEvent
})

Le guard applique aussi Anticheat.Data.eventGuard (montants abusifs sur certains events d'inventaire) et netOnlyEvents (events qui ne peuvent venir que du réseau, jamais d'un autre script serveur). Les events VORP listés dans Anticheat.Data.forbiddenEvents sont surveillés par guard_events.lua : tout déclenchement vaut sanction forceBan.

Wallhack (tir à travers les murs) ​

guard_wallhack.lua : sur weaponDamageEvent, le serveur demande à la victime de vérifier la ligne de vue vers l'attaquant (lo:anticheat:wallcheck:doCheck). La victime (wallhack.lua client) lance un shape-test entre l'attaquant et elle-même, à une distance comprise entre minDistance et maxDistance. Si un mur est touché, elle remonte lo:anticheat:wallcheck:report. Après wallhack.strikes strikes sur wallhack.window, sanction.

lua
wallhack = { enabled = true, strikes = 8, window = 120000, oncePerSession = true,
             minDistance = 10.0, maxDistance = 100.0 }

Heartbeat ​

Le client (heartbeat.lua) envoie lo:anticheat:hb (coords + état vivant) toutes les heartbeat.checkInterval ms. Le serveur (report.lua) sanctionne l'absence prolongée de heartbeat (client anticheat désactivé) après heartbeat.strikes cycles dépassant heartbeat.timeout. Les coords du heartbeat alimentent le contrôle de téléportation serveur (Anticheat.FeedHeartbeatCoords).

Limite assumée de lo_protected

lo_protected étant posé côté client, un cheateur peut le falsifier sur ses propres entités. Les modèles de crash et blacklistés restent supprimés quoi qu'il arrive : le flag ne relâche que l'heuristique de comptage. Activer entity.deleteUnprotected uniquement après avoir tagué toutes les ressources de spawn légitimes.

Protéger une ressource externe ​

Une ressource légitime qui spawn beaucoup d'entités (chevaux, props, véhicules) doit taguer ses entités pour éviter leur suppression par le watchdog et le sweep client. Le pont @lo/bridge/ac_protect.lua wrappe les natives de création dans le contexte Lua de la ressource et pose lo_protected = true sur chaque entité, sans dépendre du runtime de lo.

Ajouter dans le fxmanifest.lua de la ressource externe, en tête des client_scripts :

lua
client_scripts {
  "@lo/bridge/ac_protect.lua",  -- protège les entités spawn par cette ressource
  -- ... le reste des scripts de la ressource
}

Les natives wrappées : CreatePed, CreateObject, CreateObjectNoOffset, CreateVehicle, CreateMissionTrain, CreatePedInsideVehicle.

Pièges de chargement

  • lo doit démarrer avant la ressource externe dans server.cfg pour que l'include @lo/... se résolve.
  • Tout fichier servi via @lo/... doit être listé dans le files {} du fxmanifest de lo (ac_protect.lua y est déjà). Sinon : Failed to load script @lo/bridge/ac_protect.lua.

Côté serveur, vérifier qu'une entité est protégée :

lua
if Anticheat.IsEntityProtected(entity) then
    -- entité taguée lo_protected : ignorée par les gardes d'entités
end
