Cartes intérieures (imaps) ​

Le module imaps gère le chargement et le déchargement dynamique des cartes intérieures (interior maps / IPL) du jeu. Il maintient côté serveur un état partagé des IPL actifs, le synchronise vers tous les clients via un StateBag global, et persiste cet état dans le fichier data/imap.json. Un menu staff permet d'activer ou de désactiver à chaud n'importe quelle carte intérieure à proximité, et un jeu d'overrides force le chargement de versions custom au démarrage de la session.

Le module est réparti en trois couches :

CoucheFichierRôle
sharedmodules/imaps/shared/overrides.luaListe des IPL à retirer et à charger au spawn (Core.ImapOverrides)
servermodules/imaps/server/main.luaChargement de data/imap.json, état GlobalState.activeImaps, sauvegarde, sécurité
clientmodules/imaps/client/main.luaApplication des IPL via natives, menu staff, réconciliation

Format de données : data/imap.json ​

L'état des cartes intérieures est stocké dans data/imap.json à la racine de la ressource. C'est un tableau JSON dont chaque entrée décrit une carte intérieure.

json
[
    {
        "dec_hash": -1378517036,
        "hashname": "a_03_",
        "heading": 363.2772827148,
        "x": -3328.0,
        "y": -6400.0,
        "z": 107.0013961792,
        "active": true
    }
]

Champs lus par le module :

ChampTypeDescription
dec_hashnumberHash décimal (signé) de l'IPL. Sert de clé d'identification dans tout le module.
hashnamestringNom lisible de l'IPL (utilisé comme libellé dans le menu).
x, y, znumberCoordonnées du centre de la carte intérieure. Par défaut 0 si absent.
hnumberCap (heading). Par défaut 0 si absent.
activebooleanÉtat souhaité : true charge l'IPL, false/absent ne le charge pas.

INFO

À la lecture du fichier, le serveur construit deux tables internes : allImaps (toutes les entrées, indexées par dec_hash) et activeImaps (uniquement les entrées dont active == true). Seule activeImaps est publiée dans le StateBag global.

WARNING

Le champ de cap est nommé heading dans le fichier data/imap.json, mais le module le relit sous la clé h (imap.h or 0). Le cap n'est de toute façon pas utilisé par le chargement des IPL côté client ; il n'a qu'une valeur informative.

Chargement côté serveur ​

Au démarrage de la ressource, le serveur attend brièvement puis lit data/imap.json et publie l'état actif :

lua
CreateThread(function()
    Wait(1000)
    loadImapsFromJson()
end)

loadImapsFromJson décode le fichier, remplit allImaps / activeImaps, expose l'état actif dans GlobalState.activeImaps, puis journalise le total :

lua
GlobalState.activeImaps = activeImaps
Core.Logger.info('Imaps', '%s imaps chargés, %s actifs', total, active)

En cas de fichier vide, introuvable ou non décodable, le chargement échoue proprement et journalise une erreur via Core.Logger.error('Imaps', ...).

Synchronisation et application côté client ​

Le client applique les IPL à l'aide des natives RequestImap (chargement) et RemoveImap (déchargement), et garde une trace des IPL réellement appliqués pour éviter les doublons :

lua
local function setImap(hashNum, active)
    if not hashNum then return end
    if active then
        if not appliedImaps[hashNum] then
            RequestImap(hashNum)
            appliedImaps[hashNum] = true
        end
    elseif appliedImaps[hashNum] then
        RemoveImap(hashNum)
        appliedImaps[hashNum] = nil
    end
end

Réconciliation depuis le StateBag ​

Le client écoute les changements de GlobalState.activeImaps et réconcilie l'ensemble des IPL : il charge tout ce qui est voulu et décharge tout ce qui ne l'est plus.

lua
AddStateBagChangeHandler('activeImaps', 'global', function(_, _, value)
    if type(value) ~= 'table' then return end
    reconcile(value)
end)

La fonction reconcile parcourt l'état voulu (clés converties en nombre via tonumber), active chaque IPL voulu, puis désactive tout IPL appliqué qui n'apparaît plus dans l'état.

Application unitaire ​

Pour un changement isolé (déclenché par une action staff), le serveur diffuse l'event lo:imaps:apply qui agit sur un seul IPL sans relire tout l'état :

lua
RegisterNetEvent('lo:imaps:apply', function(hash, active)
    setImap(tonumber(hash), active == true)
end)

Overrides au spawn ​

Au démarrage du client (après un délai de 2 s), le module applique les overrides définis dans Core.ImapOverrides : il retire d'abord les IPL d'origine, puis demande les versions custom, et termine par une réconciliation complète de l'état global.

lua
CreateThread(function()
    Wait(2000)
    for _, name in ipairs(Core.ImapOverrides.remove) do
        RemoveImap(joaat(name))
    end
    for _, name in ipairs(Core.ImapOverrides.request) do
        setImap(joaat(name), true)
    end
    reconcile(GlobalState.activeImaps)
end)

Core.ImapOverrides (défini dans la couche shared) expose deux listes de noms d'IPL :

CléTypeDescription
removestring[]Noms d'IPL retirés au spawn via RemoveImap(joaat(name)).
requeststring[]Noms d'IPL chargés au spawn via RequestImap(joaat(name)).

TIP

Les overrides sont utiles pour remplacer un décor d'origine par une variante éditée. Dans la configuration livrée, les versions d'origine rho_01_, rho_01__strm_*, rho_03__strm_1, etc. sont retirées au profit de leurs équivalents *_edit.

Sécurité et permissions ​

Toutes les actions de modification sont réservées au staff de niveau 40. Le contrôle s'appuie sur la permission nommée imaps.

lua
local function isImapAdmin(source)
    return Core.Permissions.HasCommandLevel(source, 'imaps')
end

La permission imaps est associée au niveau 40 dans la configuration du framework (Config.Permissions), aux côtés de command.imaps. Tout joueur de niveau inférieur reçoit nil sur les callbacks protégés et voit ses events refusés et journalisés.

Callbacks serveur ​

CallbackRetourDescription
lo:imaps:isAdminbooleanIndique si la source dispose de la permission imaps.
lo:imaps:getAlltable ou nilRenvoie la table allImaps (toutes les cartes connues, indexées par dec_hash). Renvoie nil si la source n'est pas admin.

Exemple d'appel côté client (déclenche le callback serveur de façon synchrone) :

lua
local allImaps = Core.Callback.triggerServer('lo:imaps:getAll')
if not allImaps then
    Core.Notify('Accès refusé', 'error')
    return
end

Events serveur sécurisés ​

Les deux events de mutation sont enregistrés via Core.SecureEvent (avec limitation de débit) et vérifient systématiquement la permission imaps.

lo:imaps:setActive ​

Bascule l'état d'un IPL en mémoire et le diffuse immédiatement à tous les clients via lo:imaps:apply. Ne persiste pas dans le fichier.

lua
Core.SecureEvent('lo:imaps:setActive', function(hash, active)
    -- ...vérification permission + existence dans allImaps...
    activeImaps[hash] = active or nil
    GlobalState.activeImaps = activeImaps
    TriggerClientEvent('lo:imaps:apply', -1, hash, active == true)
end, { maxPerSecond = 20 })
ParamètreTypeDescription
hashnumberdec_hash de l'IPL ciblé. Ignoré s'il n'existe pas dans allImaps.
activebooleantrue active l'IPL, false le désactive.

Limite : maxPerSecond = 20.

lo:imaps:save ​

Persiste un lot de changements dans data/imap.json (champ active de chaque entrée) et met à jour l'état mémoire.

lua
Core.SecureEvent('lo:imaps:save', function(changes)
    -- ...vérification permission...
    for hash, active in pairs(changes) do
        if allImaps[hash] then
            allImaps[hash].active = active == true
            activeImaps[hash] = active == true or nil
            saveImapToFile(hash, active == true)
        end
    end
    GlobalState.activeImaps = activeImaps
end, { maxPerSecond = 5 })
ParamètreTypeDescription
changestableTable dec_hash -> boolean des nouveaux états à persister. Ignorée si vide.

Limite : maxPerSecond = 5. La persistance utilise SaveResourceFile(..., 'data/imap.json', json.encode(data, { indent = true }), -1).

WARNING

lo:imaps:setActive est un changement volatil (mémoire + diffusion immédiate), tandis que lo:imaps:save écrit sur disque. Tant qu'aucune sauvegarde n'a eu lieu, un redémarrage de la ressource recharge l'état du fichier et perd les bascules non sauvegardées.

Commandes staff ​

Le module enregistre deux commandes client. Le contrôle d'accès réel est effectué côté serveur lors des callbacks et events (un joueur non staff peut taper la commande mais ses actions seront refusées).

/imapmenu ​

Ouvre le menu de gestion des cartes intérieures. Le menu récupère toutes les cartes via le callback lo:imaps:getAll, calcule la distance de chaque carte au joueur et n'affiche que celles situées dans un rayon de 300 m, triées par distance croissante (limité à 200 entrées).

lua
RegisterCommand('imapmenu', function()
    openImapMenu()
end, false)

Comportement du menu :

  • Chaque entrée affiche un préfixe [ON] / [OFF], le hashname (ou IMAP_<hash> à défaut), le hash et la distance.
  • Cliquer sur une entrée bascule son état localement, enregistre le changement dans pendingChanges, et déclenche lo:imaps:setActive pour un effet immédiat sur tous les joueurs.
  • À la fermeture du menu (onBack), s'il existe des changements en attente, ils sont envoyés en bloc via lo:imaps:save pour persistance, puis une notification de succès est affichée.

/fiximap [rayon] ​

Recharge (via RequestImap) les IPL qui devraient être actifs mais ne le sont pas, dans un rayon donné. Utile pour corriger un décor intérieur qui n'aurait pas été appliqué.

lua
RegisterCommand('fiximap', function(_, args)
    local radius = tonumber(args and args[1]) or 300.0
    -- ...parcours de GlobalState.activeImaps...
    if hashNum and not IsImapActive(hashNum) then
        RequestImap(hashNum)
        fixed = fixed + 1
    end
end, false)
ArgumentTypeDéfautDescription
rayonnumber300.0Distance maximale (en mètres) autour du joueur dans laquelle corriger les IPL manquants.

À la fin, une notification résume le nombre d'IPL corrigés : Fix imaps : <n> corrigé(s) dans <rayon> m.
