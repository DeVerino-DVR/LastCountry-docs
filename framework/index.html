Introduction à lo ​

lo est le framework RedM monolithique de Last Country. C'est une ressource unique qui agrège le noyau (sessions, comptes, joueurs, personnages) et l'ensemble de ses systèmes sous un seul objet global : Core. Conçu et testé exclusivement pour RDR3 / RedM, il ne vise jamais FiveM. Le code est organisé en domaines plats, chaque gros système vit dans son propre dossier de domaine, et toute primitive de bas niveau passe par un wrapper Core.* cohérent.

Cette page présente l'identité du framework (version 2.0.0, refonte classes/services), la carte de son dépôt, le principe de structure par domaine, sa fondation jo_libs et la règle des wrappers obligatoires. Les pages suivantes détaillent l'installation, l'architecture, la configuration et l'API.

Ce qu'est lo ​

Ressource unique, objet global Core

Tout le framework tient dans une seule ressource nommée lo. Tous les domaines s'exposent sous le même objet global Core. Il n'existe pas de noyau séparé à démarrer à côté.

Caractéristiques structurantes :

  • Monolithique : un seul fxmanifest.lua, un seul point de démarrage (init.lua), une seule arborescence de configuration (config/*.lua). Chaque domaine s'accroche au global Core ; le couplage inter-domaine passe par des events lo:<côté>:<domaine>:<action> et par les hooks de cycle de vie.
  • Cible RDR3 uniquement : le manifeste déclare game 'rdr3'. Le code utilise des natives et des comportements propres à RedM.
  • Serveur-autoritaire : le cœur du framework est la Session, une machine à états par joueur possédée par le serveur. Le client ne décide jamais de son propre état ; il exécute les phases que le serveur lui ordonne.
  • Architecture classes/services : les systèmes sont des classes (shared/class.lua, helper class()), instanciées et accrochées au global Core (par exemple Core.Lottery, Core.Vip, Core.Inventory, Core.Player).

Le manifeste déclare l'identité et les capacités fournies de la ressource :

lua
fx_version 'cerulean'
game 'rdr3'
name 'lo'
version '2.0.0'
provide 'rsg-core'
provide 'rsg-inventory'

lo remplace RSGCore et son inventaire : il les fournit (provide) aux ressources tierces, qui voient RSGCore / RSGInventory via les bridges (bridge/). Il ne faut donc jamais démarrer rsg-core en parallèle.

Architecture v2 — ce qui n'existe plus

La refonte classes/services a supprimé plusieurs systèmes de l'ancienne architecture (v1) : pas de registre de modules (Core.RegisterModule), pas d'éditeur in-game (/editor, editor_overrides), pas de Core.World.Add / Core.Action.register, pas de layout core/ + modules/. La configuration staff passe désormais par la ressource externe lo_jobscreator (panels Core.PanelHost), et non par un éditeur intégré.

Carte du dépôt ​

La structure du dépôt est plate par domaine : chaque dossier de premier niveau est un domaine, avec ses sous-dossiers server/, client/, shared/.

resources/[core]/lo/
├── fxmanifest.lua          source de vérité de l'ordre de chargement (pas de globs auto)
├── init.lua                naissance de Core + chargement NUI (jo.nui.load vers lo_uiv2)
├── shared/                 class.lua, logger, utils, enums, event, callback, command,
│                           scheduler, lifecycle, jobs, money, trace
├── config/                 config.lua + une feuille par domaine (défauts de la table Config)
├── account/                {server} comptes (fivem:), bans
├── session/                {server,client} machine à états, file d'attente, spawn, boot
├── player/                 {server} classe LoPlayer + service Core.Player + save
├── character/              {server,client} CRUD personnages, créateur, skin/preview/select
├── inventory/              {server,client,shared} Core.Inventory : items, shops, craft, drops
├── items/                  {server} cache items single-source (table items en DB)
├── jobs/                   {server} métiers et sociétés
├── vip/  conditions/       {server} tiers VIP (Discord) et grants (slots/poids)
├── permissions/            {server} niveaux staff
├── panels/                 {server} panels staff exposés à lo_jobscreator
├── bridge/                 {server,client} compat RSG + dispatchers panelhost/typeconfig
├── world/                  {server,client} primitives Core.Ped/Prop/Blip/Marker/Interact…
├── hud/  voice/  notify/   {server,client} besoins, voix de proximité, notifications
├── net/  ui/  input/       réseau, menus/prompts/input, claviers
├── admin/  anticheat/      outils staff et couches de protection
├── death/ jail/ doors/     systèmes de jeu (mort, prison, portes…)
├── business/ police/       commerces, métier police
├── lockpicking/ gift/      crochetage, dons
├── vipstore/ lottery/      boutique VIP, loterie
├── emotes/ imaps/ pause/   émotes, objets de carte, menu pause
├── spawn/                  point d'entrée unique de spawn (client)
└── activities/             {bee, foliage, hunting} activités de récolte/chasse

Le front React n'est pas dans cette ressource : il vit dans la ressource séparée lo_uiv2 (resources/[ui]/lo_uiv2/web/), et lo le charge via le NUI de jo_libs (jo.nui.load('lo', 'nui://lo_uiv2/web/build/index.html')).

Ordre de chargement explicite

Le fxmanifest.lua liste chaque fichier dans l'ordre voulu — aucun glob auto, aucun préfixe aaa_/zzz_. Un fichier qui étend une table doit être listé après celui qui la crée (par exemple items/server/items.lua avant les domaines qui déclarent des items, bridge/server/panelhost.lua avant les panels/server/*).

Structure par domaine — où va le code ​

La structure par domaine est la colonne vertébrale du framework.

  • Tout gros système est un domaine : il vit dans <domaine>/{server,client,shared} et expose son service sur Core. Il n'y a ni dossier core/ ni dossier modules/, ni registre central : ajouter un domaine = créer le dossier et l'ajouter au manifeste dans le bon ordre.
  • Les services sont des classes : class() (de shared/class.lua) crée la classe, qu'on instancie et accroche au global, par exemple Core.Lottery = LotteryService().
  • Le couplage inter-domaine passe par les events lo:Server:<domaine>:<action> / lo:Client:<domaine>:<action> et par les hooks de cycle de vie (Core.Ready, Core.OnPlayerLoaded, Core.OnPlayerUnloaded, Core.OnPlayerSave) — jamais par des globales croisées non documentées.

Aucune liste en dur dans la logique

Les données statiques (libellés, catalogues, hashes, teintes) ne vivent jamais au milieu d'un fichier de logique. Elles ont des emplacements canoniques : <domaine>/shared/data/ (partagé), <domaine>/client/data/ (client), ou config/ pour les réglages numériques.

La fondation : jo_libs ​

lo repose sur la bibliothèque jo_libs comme fondation obligatoire, sans fallback ni système parallèle. Le manifeste charge la lib et déclare les modules dont le framework a besoin :

lua
shared_script {
    '@jo_libs/init.lua',
    'shared/class.lua',
    'init.lua',
    -- config/*, shared/*, ...
}

jo_lib 'player'
jo_lib 'database'
jo_lib 'component'
jo_lib 'ped-texture'
jo_lib 'menu'
jo_lib 'input'
jo_lib 'prompt-nui'
jo_lib 'callback'
jo_lib 'nui'
-- ...

Toute primitive de bas niveau passe par jo_libs : jo.callback, jo.database, jo.menu, jo.input, jo.promptNui, jo.component, jo.pedTexture, jo.player, jo.nui. La règle pratique : si jo_libs le fait, on l'utilise ; s'il le fait à moitié, on construit autour ; sinon seulement on implémente, dans le style de la lib.

Vérifier l'API réelle dans le code de la lib

La documentation en ligne de jo_libs est parfois inexacte, et plusieurs modules ont un nom de dossier différent de leur usage (ped-texture, prompt-nui). Toujours vérifier l'API réelle dans [lib]/jo_libs/modules/<module>/.

Jamais jo.framework / jo.fw dans le code de lo

Tout passe par NOTRE noyau et NOTRE inventaire (Core.Player:Get(src), Core.Inventory.*, statebags userId/charId). Le framework-bridge de jo_libs détecte rsg-core, qui est notre propre provide : l'utiliser depuis lo est un aller-retour absurde qui casse le démarrage. Les bridges ne servent qu'aux ressources externes.

Le principe des wrappers Core.* ​

C'est l'un des principes les plus stricts du framework : toute primitive jo_libs et toute primitive monde possède un wrapper Core.*, et le code n'appelle jamais le jo.* brut ni la native brute — uniquement le wrapper. C'est ce qui rend le framework cohérent, débogable et réutilisable d'un domaine à l'autre.

À utiliserJamais
Core.Notify(msg, kind, dur) / Core.Notify(src, msg, kind, dur) (serveur)jo.notif.*
Core.Callback.register / .triggerServer / .triggerClientjo.callback.*
Core.Menu.*jo.menu.*
Core.Prompt.createNative / .createTarget / .createFollow / .createSimplejo.prompt.*
Core.Input.text / .number / .select / .confirm / .formjo.input.*
Core.RegisterKey(...) / Core.Keys.*RegisterKeyMapping
Core.RegisterCommand(name, fn, { level, suggestion })RegisterCommand
Core.Ped / Prop / Vehicle / Horse / Blip / Marker / Text3D / InteractCreatePed / CreateObject… bruts
Core.SafeTriggerClientEvent(name, target, ...)TriggerClientEvent(..., -1, ...) direct
Core.Scheduler.Register(name, interval, fn)boucle while true do Wait() ad hoc

Les wrappers Core.Callback et Core.Menu sont des proxies transparents (setmetatable({}, { __index = jo.callback / jo.menu })) : l'API sous-jacente est identique, mais l'indirection est obligatoire.

Conséquences à retenir :

  • Le serveur ne crée jamais d'entité : seul le client spawn, via les primitives Core.Ped / Core.Prop / Core.Vehicle (qui gèrent streaming, nettoyage et cohérence). Jamais de native brute dans la logique.
  • Ne jamais redéclarer des touches en local : utiliser Core.RegisterKey(name, desc, defaultKey) / Core.Keys.*, jamais RegisterKeyMapping.
  • Tout broadcast -1 passe par Core.SafeTriggerClientEvent (latent si > ~15 Ko, sinon kick 16 Ko de tout le serveur).
lua
Core.RegisterKey('myaction', 'Mon action', 'F5'):onPressed(function()
    Core.Notify('Action déclenchée', 'success')
end)
