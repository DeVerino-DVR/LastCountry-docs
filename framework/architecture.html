Architecture ​

Le framework lo (objet global Core, version 2.0.0, refonte classes/services) est organisé autour d'un concept central : la Session, une machine à états par joueur dont le serveur est l'unique propriétaire. Cette page décrit la structure par domaine, l'ordre de chargement, le boot et le cycle de vie, la machine à états de session, l'objet LoPlayer serveur, le schéma de base de données et le spawner unique.

Structure plate par domaine ​

Il n'y a ni dossier core/ ni dossier modules/. Chaque dossier de premier niveau est un domaine avec ses sous-dossiers server/, client/, shared/. Les systèmes sont des classes (shared/class.lua, helper class()), instanciées et accrochées au global Core :

lua
local LotteryService = class()

function LotteryService:constructor()
    -- ...
end

Core.Lottery = LotteryService()

Il n'existe aucun registre central de domaines. Un domaine s'expose simplement sur Core au chargement de son fichier ; l'ordre est garanti par le manifeste, et la disponibilité runtime par les hooks de cycle de vie.

Ordre de chargement ​

Le fxmanifest.lua est la source de vérité de l'ordre.

Aucun glob automatique global, aucun préfixe aaa_/zzz_. Chaque fichier est listé explicitement dans l'ordre voulu : un fichier qui étend une table est listé après celui qui la crée.

Points clés de l'ordre :

  • shared : @jo_libs/init.lua → shared/class.lua → init.lua → config/*.lua → shared/* (logger, utils, enums, event, callback, command, scheduler, lifecycle…).
  • server : database/server/schema.lua en premier, puis account / permissions, puis bridge/server/panelhost.lua + typeconfighost.lua (avant les panels/server/*), puis player / session, puis items/server/items.lua (avant les domaines qui déclarent des items), puis le reste, puis bridge/server/* (compat) et anticheat/server/*.
  • client : world/client/* (primitives) tôt, puis notify / hud / ui / input, puis session / character, puis les domaines de jeu, puis spawn / session/client/boot.lua, enfin bridge/client/* et anticheat/client/*.

Boot et cycle de vie ​

Core est créé dans init.lua. Les hooks de cycle de vie (shared/lifecycle.lua, shared/event.lua) remplacent les events bruts du runtime et sont disponibles dès le chargement de n'importe quel fichier :

lua
Core.Ready(function() end)            -- framework prêt (DB, cache items chargés)
Core.OnPlayerLoaded(function(player) end)    -- perso chargé et actif
Core.OnPlayerUnloaded(function(player) end)  -- joueur déchargé (déconnexion, switch)
Core.OnPlayerSave(function(player) end)      -- à chaque sauvegarde du joueur

Jamais de MySQL.*.await au top-level d'un fichier

Un appel DB bloquant au chargement avorte le chunk, et le service du domaine n'est jamais exposé sur Core. Tout accès DB et toute lecture de config différée vont dans Core.Ready.

C'est aussi dans Core.Ready qu'un domaine déclare les items dont il a besoin et charge sa configuration de panel.

La Session : machine à états serveur-autoritaire ​

Le client ne décide jamais de son propre état. Il exécute des phases que le serveur lui ordonne et notifie leur complétion. La machine à états est définie dans shared/enums.lua (Core.Enums.SessionState) et validée par session/server/manager.lua.

text
CONNECTING → IDENTIFIED → ACCOUNT_LOADED → CHARACTER_SELECT ⟷ CREATOR
   → CHARACTER_LOADING → PRELOADING → SPAWNING → ACTIVE → DROPPING
  • CONNECTING → IDENTIFIED : identification du compte par fivem: (compte Cfx.re).
  • ACCOUNT_LOADED : users chargé (SELECT/INSERT), vérification de ban.
  • CHARACTER_SELECT ⟷ CREATOR : sélection de personnage ou entrée dans le créateur (bucket isolé).
  • CHARACTER_LOADING → PRELOADING → SPAWNING : construction du LoPlayer, préchargement (skin, scène), passage par le spawner unique.
  • ACTIVE : session active, bucket 0, Core.OnPlayerLoaded déclenché.
  • DROPPING : déconnexion, sauvegarde et logout.

Pourquoi serveur-autoritaire

  • Un seul endroit décide quand un joueur spawn : pas de logiques de spawn divergentes.
  • Le restart de la ressource en développement est trivial : le serveur reconstruit les sessions et rejoue les phases.
  • Les bridges peuvent émettre les events RSG attendus au moment exact du cycle.

Identité et ciblage ​

L'identité d'un compte est le fivem: (compte Cfx.re). Les autres identifiants (license, steam) restent stockés pour l'anticheat, jamais comme clé de compte. Toujours cibler par userId (compte) ou charId (personnage), stables entre les sessions — jamais par source, qui change à chaque connexion.

Objet Player (serveur) ​

L'objet Player runtime est une instance de la classe LoPlayer (player/server/player.lua), récupérable via Core.Players[src] ou Core.Player:Get(src).

lua
local player = Core.Player:Get(src)

player.source          -- session courante (interne, jamais comme clé de ciblage)
player.userId          -- compte (stable)
player.charId          -- personnage (stable)
player.PlayerData      -- { citizenid, charinfo, money, job, gang, metadata, position }
player.Functions       -- HasItem(name, n), RemoveItem(name, n)…

player:GetMoney(t)
player:AddMoney(t, n, reason)
player:RemoveMoney(t, n, reason)
player:SetMoney(t, n, reason)
player:SetJob(name, grade)
player:SetGang(name, grade)
player:SetMetaData(k, v)
player:GetMetaData(k)
player:PublishState()  -- réplique vers Player(src).state

Le service Core.Player (player/server/service.lua) fournit l'accès et le cycle de vie :

lua
Core.Player:Get(src)
Core.Player:GetByCitizenId(cid)
Core.Player:GetByUserId(userId)
Core.Player:GetByCharId(charId)
Core.Player:GetAll()
Core.Player:Login(session, charId)
Core.Player:Logout(src)
Core.Player:Save(src)

StateBags répliquées ​

Plutôt que des net events, l'état d'un joueur est répliqué via statebags (Player(src).state.*) : userId, charId, citizenid, job, gang, money, charinfo, onduty.

StateBag plutôt qu'events

Pour synchroniser un état, préférer GlobalState / Player(src).state + AddStateBagChangeHandler aux net events : plus sûr et plus optimisé. Les statebags survivent aussi au restart de la ressource, ce qui facilite la reconstruction des sessions en développement.

Events réseau ​

  • Client → Serveur : TriggerServerEvent('lo:Server:<domaine>:<action>'), reçu par RegisterNetEvent.
  • Serveur → Client : TriggerClientEvent('lo:Client:<domaine>:<action>', src, ...), reçu par RegisterNetEvent.
  • Tout event qui traverse le réseau se reçoit avec RegisterNetEvent, jamais AddEventHandler (sinon event X was not safe for net). AddEventHandler ne sert que pour un TriggerEvent local.
  • Aller-retour : Core.Callback, jamais d'events manuels.
  • Broadcast -1 : Core.SafeTriggerClientEvent.

Helpers de cycle de vie ​

shared/command.lua, shared/lifecycle.lua, shared/scheduler.lua exposent des helpers disponibles des deux côtés.

Core.RegisterCommand ​

Core.RegisterCommand(name, fn, opts) enregistre une commande. Le callback fn(source, args, level, raw) reçoit le niveau staff résolu. L'option opts.level (un nombre ou une clé de Config.Perms) refuse automatiquement l'exécution sous ce niveau.

lua
Core.RegisterCommand('goto', function(source, args, level, raw)
    local charId = tonumber(args[1])
    -- ...
end, { level = 'goto' })

Core.Scheduler ​

Core.Scheduler.Register(name, interval, fn) remplace les boucles while true do Wait() end ad hoc par une boucle nommée et traçable.

Schéma de base de données ​

Les tables sont créées et migrées (idempotent) par database/server/schema.lua, via jo.database.addTable et jo.database.addColumn :

users, characters, inventories, items, jobs, gangs, societies, lo_config, lo_vip_houses, lo_jail, lo_businesses, lo_business_grades, lo_business_employers, ac_detections, ac_ban_identity.

sql
users      (compte, indexé par fivem: / Cfx.re ; license, steam stockés pour l'anticheat)
characters (personnages, indexés par citizenid ; charinfo/money/job/gang/metadata/position JSON,
            skin/clothes ; soft delete via deleted_at)
items      (catalogue single-source des items, chargé en cache au boot)
lo_config  (overrides de config persistés par Core.ConfigStore)

Migrations

Les migrations passent exclusivement par jo.database.addTable / addColumn. La suppression d'un personnage est un soft delete : la colonne deleted_at est renseignée, la ligne n'est jamais effacée.

Items single-source ​

Les items sont stockés en DB (table items) et chargés en cache au boot par items/server/items.lua (Core.Items:Get / :GetAll / :Upsert / :Remove). Chaque domaine déclare inline les items dont il a besoin via Core.Items:Ensure(list) dans son Core.Ready — insertion si absent (jamais d'écrasement). Pas de fichier central d'items.

lua
Core.Ready(function()
    Core.Items:Ensure({
        { name = 'mon_item', label = 'Mon item', weight = 10, description = '…', useButton = 'Utiliser' },
    })
end)

Spawner unique (client) ​

Un seul point d'entrée existe pour tous les chemins de spawn (spawn/client/spawner.lua). La raison du spawn est passée explicitement et conditionne la séquence (fade, chargement modèle, résurrection, application skin/clothes, chargement de collision et de scène, unfreeze).

lua
Spawner.spawn({
    reason = 'first',   -- 'first' | 'switch' | 'creator' | 'respawn' | 'restart'
    coords = vector4,
    model  = hash,
    skin   = {},
    clothes = {},
})

Pas de boucle de surveillance

Aucune boucle while true qui surveille l'état de spawn n'est admise. Le spawner est appelé, il termine ou échoue avec timeout via jo.waiter. Rien d'autre.

Config et panels staff ​

config/config.lua + config/*.lua = défauts (la table Config). Ne jamais capturer un scalaire Config en local au chargement : capturer la table parente ou lire au point d'usage.

Les overrides sont persistés en DB (table lo_config) via Core.ConfigStore (bridge/server/panelhost.lua) :

lua
local data = Core.ConfigStore.Load(id, migrateFile)   -- DB, fallback fichier JSON
Core.ConfigStore.Save(id, tbl)

La configuration éditable en jeu passe par la ressource externe lo_jobscreator (et non par un éditeur intégré). Côté lo, deux dispatchers partagés possèdent les exports uniques par ressource :

  • bridge/server/panelhost.lua → Core.PanelHost + exports JcGetPanelConfig / JcSavePanelConfig / JcPanelAction.
  • bridge/server/typeconfighost.lua → Core.TypeConfig + exports JcGetTypeConfig / JcSaveTypeConfig.

Un panel s'enregistre via Core.PanelHost.Define{ id, label, icon, section, schema, root = fn, … } (charger la config dans Core.Ready). Voir panels/server/*.lua.
