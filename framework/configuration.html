Configuration (Config) ​

La table globale Config est la source unique des défauts du framework lo. Elle est créée dans config/config.lua puis enrichie par les autres feuilles de config/*.lua, et exposée via _ENV.Config. Chaque feuille couvre un domaine : Config.Game, Config.Inventory, Config.Hud, Config.Anticheat, etc. Tout réglage du framework — distances, délais, niveaux de permission, seuils anticheat — vit dans ces fichiers.

Feuilles de Config

config/config.lua porte Config.Debug, Config.Game, Config.Queue, Config.Vip, Config.Perms, Config.Performance, Config.BlipGrouping. Les autres feuilles sont des fichiers dédiés dans config/ : inventory.lua, creator.lua, hud.lua, voice.lua, admin.lua, anticheat.lua, bee.lua, gift.lua, vipstore.lua, lottery.lua, weapon.lua, foliage.lua, jail.lua, police.lua, business.lua, doors.lua, emotes.lua, hunting.lua. L'ordre de chargement est explicite dans fxmanifest.lua : une feuille qui étend Config est listée après celle qui crée la table.

Règle critique : ne jamais capturer un scalaire Config en local

Ne jamais copier un scalaire de Config dans une variable locale au chargement — la valeur ne suivra plus les overrides appliqués à chaud :

lua
-- MAUVAIS : la valeur ne suivra plus les overrides
local dist = Config.Inventory.dropDistance

-- BON : capturer la table parente (la mutation reste visible)
local inv = Config.Inventory
-- ... puis lire inv.dropDistance au point d'usage

-- BON aussi : lire directement au point d'usage
local d = Config.Inventory.dropDistance

Les panels mutent la valeur dans la table Config (container[key] = value). Un scalaire copié à part garde l'ancienne valeur. Capturez la table parente, ou lisez au point d'usage.

Overrides persistés (Core.ConfigStore) ​

config/*.lua ne contient que les défauts. Les valeurs modifiées en jeu par le staff sont persistées en base, dans la table lo_config, via Core.ConfigStore (bridge/server/panelhost.lua). Chaque jeu de réglages est stocké sous un id (le data est un JSON).

lua
local data = Core.ConfigStore.Load(id, migrateFile)
Core.ConfigStore.Save(id, tbl)
  • Core.ConfigStore.Load(id, migrateFile) lit la ligne lo_config correspondant à id. Si la DB est vide et qu'un ancien fichier JSON (migrateFile, ex. data/vip_overrides.json) existe, il est importé une fois puis écrit en base. Renvoie nil si rien n'est trouvé.
  • Core.ConfigStore.Save(id, tbl) sérialise tbl en JSON et l'upsert dans lo_config.

Au boot, chaque panel charge ses overrides et les applique par-dessus les défauts (voir ci-dessous). Le résultat : config/*.lua reste intact, et l'état effectif vit en base.

Édition en jeu — lo_jobscreator ​

La configuration éditable en jeu par le staff passe par la ressource externe lo_jobscreator (système de panels), pas par un éditeur intégré au core. Côté lo, un panel s'enregistre via Core.PanelHost.Define{ ... } (bridge/server/panelhost.lua). À l'enregistrement (dans Core.Ready), le panel :

  1. charge ses overrides via Core.ConfigStore.Load et les applique sur la feuille Config ciblée ;
  2. se publie dans lo_jobscreator (exports.lo_jobscreator:RegisterPanel) ;
  3. expose un get / save consommés par les exports JcGetPanelConfig / JcSavePanelConfig.
lua
Core.PanelHost.Define({
    id = 'survival',
    label = 'Survie & Gameplay',
    icon = 'star',
    schema = SCHEMA,
    file = 'data/survival_overrides.json',
    root = function() return Config end,
    publish = function() GlobalState.cfgSurvival = Core.PanelHost.Collect(SCHEMA, Config) end,
})

Voir la page Éditeur de configuration pour le détail du système de panels (Core.PanelHost, Core.TypeConfig, exports).

Config.Anticheat ​

Feuille la plus volumineuse (config/anticheat.lua). Système serveur-autoritaire ; les réglages racine pilotent le comportement global (enabled, action, testMode, exemptLevel, interval, strikes…) et des sous-tables couvrent chaque domaine de détection (weapons, entity, damage, killDistance, explosion, masskill, state, connect, checks, flood, wallhack, watchdog, integrity, banEvasion, gameEvents, roleGuard…).

Comportement par défaut

Par défaut testMode = true et action = 'log' : l'anticheat journalise sans sanctionner. Passez ces réglages (via le panel anticheat) avant de compter sur des kicks/bans automatiques en production.
